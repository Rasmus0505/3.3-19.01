# 第07章：容器、Docker、镜像、端口、环境变量

## 7.1 为什么要容器

你可能遇到过这类问题：

- 在我电脑能跑，在线上跑不起来
- 依赖版本不一致
- 系统缺少某些工具（如 ffmpeg）

容器的作用就是把“运行环境 + 代码 + 依赖”一起打包，减少环境差异。

---

## 7.2 容器、镜像、Docker 三个词的区别

### Docker

- 构建和运行容器的工具链

### 镜像（Image）

- 容器的“静态模板”
- 包含代码、依赖、启动命令

### 容器（Container）

- 镜像启动后的运行实例

简化关系：

```text
Dockerfile -> 构建镜像 -> 运行成容器
```

---

## 7.3 本项目 Dockerfile 讲解

本项目是多阶段构建：

1. 前端阶段：`node:22-alpine` 构建 React 静态资源
2. 后端阶段：`python:3.11-slim` 安装依赖并复制代码
3. 最终容器启动 `scripts/start.sh`

关键点：

- 安装了 `ffmpeg`，支持音频处理
- 暴露端口 `8080`
- 启动命令最终跑 `uvicorn`

---

## 7.4 端口（Port）到底是什么

端口是服务进程监听的逻辑通道号。

本项目默认：

- 应用监听 `PORT`（默认 `8080`）

在 Zeabur 上，平台会处理公网访问与端口映射，你主要关注：

- 程序是否正确监听平台注入的 `PORT`

---

## 7.5 环境变量（Env）是什么

环境变量是运行时配置，不写死在代码里。

### 为什么必须用环境变量

1. 避免密钥泄露到仓库
2. 不同环境配置不同（开发/测试/生产）
3. 平台化部署（Zeabur）天然支持

### 本项目必要环境变量

- `DATABASE_URL`
- `DASHSCOPE_API_KEY`
- `JWT_SECRET`
- `ADMIN_EMAILS`

### 常见可选变量

- `TMP_WORK_DIR=/tmp/zeabur3.3`
- `MT_BASE_URL`
- `MT_MODEL`

---

## 7.6 启动脚本在做什么

`scripts/start.sh` 关键动作：

1. 检查关键环境变量是否存在（日志中有 `[DEBUG]`）
2. 提示当前仓库改为手动迁移模式
3. 最终启动 `uvicorn app.main:app`

数据库结构变更统一手动执行：

```text
python -m alembic -c alembic.ini upgrade head
```

---

## 7.7 新手常见误区

### 误区 1：把 key 写进源码更方便

这是高风险行为，必须避免。密钥只放环境变量。

### 误区 2：端口写死 8080 就够了

在托管平台通常应优先读取平台注入端口（本项目已支持 `PORT`）。

### 误区 3：容器等于虚拟机

容器更轻量，多个容器共享宿主机内核，运行方式和虚拟机不同。
