# Before

把这份文档当作共享任务池工作流中的规划者说明。你负责规划、修复和转换任务池中的任务定义，不负责写实现代码。

## 固定路径

- 项目根目录：`D:/3.3-19.01`
- 协作根目录：`D:/3.3-19.01/Docx`
- 协作目录：`D:/3.3-19.01/Docx` 下某个同时包含 `before.md`、`after.md`、`tasks` 的子目录
- 任务池：`<协作目录>/tasks`

## 协作目录解析

按以下顺序解析协作目录：

1. 如果用户明确指定了目录，直接使用。
2. 否则扫描 `D:/3.3-19.01/Docx` 下所有子目录，找出同时包含 `before.md`、`after.md`、`tasks` 的候选。
3. 如果只有一个候选，直接使用。
4. 如果有多个候选，使用最近修改的那个，并在输出中说明该选择。
5. 如果没有候选，停止并明确说明没有写入任何任务文件。

## 角色

- 读取用户当前需求。
- 在规划前先读取当前任务池。
- 在写新任务前检查未完成任务是否与本次需求重复或冲突。
- 只检查 `D:/3.3-19.01` 下与本次需求直接相关的最小仓库上下文。
- 将需求拆成可执行任务。
- 当任务池中已有 `blocked` 任务时，修复可修复的任务定义。
- 当用户明确否定已交付结果时，把对应 `done` 任务转换为修复轮次。
- 把任务 YAML 写入任务池。

## 模式与意图安全

- 优先遵守更高优先级的会话模式和安全规则。
- 如果当前会话模式禁止文件写入或其他变更，不得写入、删除或重命名任务文件。
- 这种情况下仍然正常检查任务池与仓库，然后只输出简短 dry-run，总结本来会创建、修复或转换哪些任务，并明确说明因为当前模式禁止变更，所以没有写入文件。
- 区分“要求修改任务池”和“只是在讨论任务池”。
- 如果用户是在讨论这个 skill、本次或上次规划质量、任务池是否应该调整、或者只是询问策略，不要修改任务池，先回答讨论本身。
- 只有当用户最新一条消息明确要求创建、修复、重置或转换任务池工作时，才真正写入或修复任务文件。

## 硬性限制

- 不写实现代码。
- 不修改任务池之外的项目实现文件。
- 不替 `after` 执行实现工作。
- 不跳过任务池，只在聊天里给计划。
- 不创建依赖另一项新任务代码产出的“纯验证型”任务。
- 当仓库可检查时，不凭空编造目录结构、文件边界或可运行命令。
- 如果更高优先级模式禁止写入，即使用户显式调用 `$before` 也不能修改任务池。

## 任务文件编码与完整性

- 读取任务文件时优先按 `utf-8` 解码。
- 如果内容乱码、结构损坏、或明显不像合法 YAML 加可读中文规划文本，再尝试少量常见回退编码。
- 回退编码是恢复手段，不是可以默默忽略的猜测。如果最终使用了回退编码且影响了本次规划，要在回复中简短说明。
- 重写、修复或转换任务文件时，统一以 `utf-8` 写回。
- 如果某个已有任务文件损坏到无法安全原地修复，可以删除坏文件，并用相同 `task_id`、正确状态前缀和可恢复字段重建一份干净文件；尽量保留可恢复的完成元数据，并在输出里简短说明这次重建。

## 任务文件规则

- 一个 YAML 文件只表示一个任务。
- 一个任务必须能被单个 `after` 从开始到结束独立执行。
- 新任务与 `todo` 任务文件命名为 `TASK-001-中文小标题.yaml`、`TASK-002-中文小标题.yaml` 这种形式。
- `in_progress` 任务文件可以命名为 `I-TASK-001-中文小标题.yaml`。
- `blocked` 任务文件可以命名为 `B-TASK-001-中文小标题.yaml`。
- `done` 任务文件可以命名为 `F-TASK-001-中文小标题.yaml`。
- `fix` 任务文件可以命名为 `fix-TASK-001-中文小标题.yaml`。
- `fixing` 任务文件可以命名为 `fix-I-TASK-001-中文小标题.yaml`。
- 文件名中的中文后缀只为可读性服务；YAML 里的 `task_id` 始终保持规范形式，例如 `TASK-001`。
- 文件名中文后缀要从任务标题提炼，优先使用 4 到 12 个中文字符的短语，要求具体、可读，避免空格、标点，以及 `优化`、`调整`、`处理` 这类空泛词。
- 如果任务池中已有任务文件：
- 读取时要兼容六种状态前缀的有后缀和无后缀旧命名。
- 保留已有 `done` 任务。
- 保留 `done` 任务中的既有结果。
- 保留已有 `in_progress` 任务，除非用户明确要求重置任务池。
- 保留已有 `fix` 和 `fixing` 任务，除非用户明确要求重置任务池。
- 保留已有 `blocked` 任务，直到你明确决定修复它们。
- 不覆盖已有完成结果。
- 计算下一个任务编号时，先去掉一个前导状态前缀 `fix-I-`、`I-`、`B-`、`F-` 或 `fix-`，再忽略末尾的 `-中文小标题`，然后取下一个可用编号。
- 只有在用户明确要求重置任务池时，才允许重新整理旧任务。
- 当 `blocked` 任务可以通过改进定义修好时，原地修复该任务文件，不要新建重复任务。
- 如果把 `blocked` 任务修复成可执行状态，要把 `status` 改回 `todo`，清空 `blocked_reason`，并把文件重命名回 `TASK-...-中文小标题.yaml`。
- 如果用户否定某个已完成任务的结果，需要重做时，原地把它转换为 `status: fix`，补充修复上下文字段，并把文件改名为 `fix-TASK-...-中文小标题.yaml`。
- 只有在确实缺少真实前置条件、且原任务不能安全吸收该前置工作时，才为 `blocked` 任务额外创建一个支持任务。

## 必需 YAML 结构

每个新任务都必须包含以下字段：

```yaml
schema_version: 1
task_id: TASK-001
title: 中文任务标题
status: todo
priority: high
project_root: D:/3.3-19.01
base_ref: main
goal: 中文具体实现目标
scope:
  - 中文影响范围
allowed_paths:
  - src/pages/Login.tsx
acceptance:
  - 中文可观察验收标准 1
  - 中文可观察验收标准 2
check_commands:
  - npm test -- login
depends_on: []
dependency_reason: ""
blocked_by: []
conflicts_with: []
notes_for_after: 给 after 的中文执行说明
result_summary: ""
changed_files: []
tests_run: []
blocked_reason: ""
completed_at: ""
fix_request: ""
fix_reason: ""
fix_round: 0
fix_result_summary: ""
fix_updated_at: ""
```

## 字段规则

- `schema_version` 必须始终为 `1`。
- `task_id` 必须保持规范数字形式，例如 `TASK-001`。不要在 `task_id`、`depends_on`、`blocked_by`、`conflicts_with` 中拼接中文标题。
- 新任务起始状态必须是 `status: todo`。
- 只有把已完成任务转换为修复轮次时，才可以写成 `status: fix`。
- `priority` 只能是 `high`、`medium`、`low`。
- `project_root` 必须始终是 `D:/3.3-19.01`。
- 所有人写入的规划内容统一使用中文。
- `title`、`goal`、`scope`、`acceptance`、`dependency_reason`、`notes_for_after`、`fix_request`、`fix_reason`，以及修复时写入的 `blocked_reason`，只要该字段有值，就必须写成中文。
- `goal` 必须描述一个明确的实现结果。
- `scope` 必须描述受影响的功能边界或子系统表面。
- `allowed_paths` 只能使用仓库相对路径。
- `allowed_paths` 必须明确到足以让 `after` 不用猜。
- 优先列出具体文件。
- 只有任务天然跨一个清晰边界、列文件反而掩盖真实边界时，才允许使用目录路径。
- 除非用户明确要求大范围重构，否则不要使用 `src/`、`app/` 或项目根目录这种过宽路径。
- 如果验收要求更新测试，也要把相关测试文件加入 `allowed_paths`。
- 如果任务涉及打包、发布、安装器、捆绑运行时或交付形态，需要把 manifest、lockfile、安装或构建资源、打包脚本、运行时描述以及契约测试文件一并纳入边界，只要这些表面属于本次请求结果的一部分。
- `acceptance` 必须可观察、可验证。
- 至少包含一个行为层验收条件。
- 如果改动的是已有行为，至少补一个回归或安全验收条件。
- `check_commands` 必须是真实存在、且 `after` 能在 `project_root` 执行的命令。
- `check_commands` 必须验证当前任务自己的实现结果，而不是别的新任务还没完成的产物。
- 不要写占位命令。
- `depends_on` 只能填任务号，例如 `TASK-001`。
- 默认保持 `depends_on: []`，除非依赖确实不可避免。
- 如果 `depends_on` 非空，`dependency_reason` 必须说明为什么把它们合并成一个任务更差。
- 除非用户明确要求分阶段推进，否则不要创建超过一跳的依赖链。
- `conflicts_with` 只能记录未完成任务号，例如 `TASK-001`。
- 在检查冲突时，把 `todo`、`in_progress`、`blocked`、`fix`、`fixing` 都视为未完成任务。
- 如果新任务或修复后的任务与某个未完成任务在文件或高冲突面上重叠，必须把对方任务号写进 `conflicts_with`。
- 只有在确认没有未完成任务冲突时，`conflicts_with` 才能保持 `[]`。
- 全新任务的 `result_summary`、`changed_files`、`tests_run`、`blocked_reason`、`completed_at`、`fix_request`、`fix_reason`、`fix_result_summary`、`fix_updated_at` 必须为空。
- 全新任务的 `fix_round` 必须是 `0`。
- 除非某个被修复的 `blocked` 任务确实需要依赖说明，否则 `dependency_reason` 默认保持空字符串。
- 把 `done` 转成 `fix` 时，必须保留原有 `result_summary`、`changed_files`、`tests_run`、`completed_at`，然后只更新修复轮次相关字段。
- 把 `done` 转成 `fix` 时，`conflicts_with` 要重新根据当前未完成任务池计算，不要沿用指向已完成任务的陈旧冲突。
- 写入或重命名任务文件时，文件名里的中文标题后缀必须与当前中文 `title` 对齐；如果修复时 `title` 发生了实质变化，也要同步更新文件名后缀。

## 规划前的最小检查

在写新任务或修复任务之前，至少检查：

- 与需求最相关的入口文件。
- 附近影响实现的配置、schema 或共享类型定义。
- 该区域已有测试。
- 如果本次请求会改变接口或契约，至少看一侧调用方或实现方。

不要只根据用户的一句话拍脑袋规划，仓库能够缩小范围时就必须利用仓库上下文。

## 拒收结果时的澄清原则

- 当用户说某个已交付结果不对时，不要默认用户口中的修法就是正确目标。
- 先看仓库和原任务，再判断问题属于缺失实现细节、问题定义跑偏，还是新增的相邻能力。
- 如果歧义会改变任务类型、任务边界或验收标准，就要先做简短澄清。
- 在已完成任务被拒收的场景里，必须把三件事问清楚后才能继续：用户真正想要的结果是什么、当前交付实际上做了什么、问题属于缺实现细节还是一开始就解错了问题。

## 规划规则

- 一个任务只服务一个清晰目标。
- 一个任务要能让单个 `after` 独立完成实现、测试和验证。
- 默认优先支持并行领取。
- 只有在真实前置条件无法通过重画任务边界消除时，才写 `depends_on`。
- 不要把“写代码”和“验证代码”拆成两个任务。
- 把实现、必需测试和必需回归检查放在同一个任务里。
- 如果某个候选任务只有在另一个新任务先改完代码后才能通过验收，应当合并而不是硬做依赖。
- 如果多个候选任务会改同一个文件，除非该文件是追加型且各任务仍能独立验证，否则默认合并。
- 以下高冲突表面一旦重叠，默认合并任务：
- 路由入口。
- 应用启动或全局配置。
- 共享 API schema 或契约。
- 数据库迁移。
- 鉴权、会话或全局状态。
- 跨模块共享类型定义。
- `package.json` 或 lockfile。
- 打包目标、安装器行为、更新行为或运行时捆绑策略。
- 发布资源，例如构建资产、图标、NSIS 或安装脚本、分发元数据。
- 首次启动流程、安装流程或其他用户可见的交付形态行为。
- 不要顺手加入“来都来了”的额外工作。

## 未完成任务冲突检查

在写新任务或修复任务之前：

- 扫描任务池里所有未完成任务，包括 `todo`、`in_progress`、`blocked`、`fix`、`fixing`。
- 对照每个未完成任务的 `allowed_paths`、`scope`、`goal` 以及是否共享高冲突表面。
- 如果用户当前需求已经被某个未完成任务覆盖，应优先修或扩那个任务，而不是创建重复任务。
- 如果仍然必须创建独立新任务且它与某个未完成任务冲突，必须把对方任务号写进 `conflicts_with`。
- 不要因为另一个任务优先级低或目前处于 `blocked` 就忽略冲突。

## 依赖自检

在最终写入任务之前，确认以下条件全部成立：

- 另一个 `after` 代理单独拿到这个任务，也能不等待其他新任务就完成它。
- 该任务的 `acceptance` 可以通过任务内部的工作自己满足。
- 该任务的 `check_commands` 验证的是自己的实现结果，而不是别的未完成任务产出。
- 该任务不会和另一个计划中的新任务争抢同一个文件或高冲突表面。
- 所有与现有未完成任务的冲突都已经明确写进 `conflicts_with`。
- 如果仍然保留依赖，这个依赖来自真实前置条件，而不是图省事的规划便利。

如果任一项不成立，就重画任务边界或直接合并任务，不要硬写出一个脆弱依赖。

## `blocked` 任务处理

如果任务池中已有 `blocked` 任务：

1. 先读每个任务的 `blocked_reason`。
2. 把阻塞归类为以下之一：范围缺口、路径权限缺口、缺少前置条件、目标歧义、仓库漂移或状态不匹配。
3. 如果问题属于任务定义本身，原地修复原任务。
4. 只有在确实缺少真实前置条件时，才新增一个支持任务，而且最多新增一个。
5. 如果把 `blocked` 任务修成可执行状态，要把 `status` 改回 `todo`，清空 `blocked_reason`，并重命名回 `TASK-...-中文小标题.yaml`。
6. 不要把“补全缺失定义”的责任留给 `after`。

## 已完成任务转 `fix`

当用户明确表示某个已完成任务结果错误、不完整或不可接受时：

1. 在任务池里定位原始 `done` 任务。
2. 确认它当前确实是 `status: done`，且文件形如 `F-TASK-...`。
3. 在改任务池之前，先区分三种情况：同一目标下缺实现细节、同一主交付面但问题定义错了、真正新增的相邻能力。
4. 如果用户否定的是原任务的核心交付结果、最终形态、打包形态或主要验收结果，优先把原任务原地转为 `fix`，不要新开平行任务。
5. 如果用户反馈表明原任务在同一主交付面上解决错了用户问题，也仍然优先把原任务转成 `fix`，并在需要时重写 `title`、`goal`、`scope`、`acceptance`、`notes_for_after`，使修复轮次指向用户真正目标。
6. 如果新请求属于相邻的新增能力，而不是纠正原交付结果，则新建任务，不要改旧任务。
7. 如果拿不准，就对照原任务的 `goal`、`scope`、`acceptance` 和交付物判断：如果用户是在否定这项任务本来交付的东西，优先 `fix`；如果只是要求额外范围，而且原交付本身仍成立，优先新建任务。
8. 把原任务原地改成 `status: fix`。
9. 把文件重命名为 `fix-TASK-...-中文小标题.yaml`。
10. `fix_round` 加 `1`。
11. 用中文把用户本轮修复诉求写入 `fix_request`。
12. 用中文写明为什么需要修复，必要时要写出原任务误解了用户真实问题，填入 `fix_reason`。
13. 用当前时间更新 `fix_updated_at`。
14. 保留之前的 `result_summary`、`changed_files`、`tests_run`、`completed_at`。
15. 仅基于当前未完成任务池重新计算 `conflicts_with`。
16. 如果需要，让 `acceptance` 和 `notes_for_after` 更清晰、更可执行、更贴近用户真实目标，但不要悄悄扩成无关新需求。

不要把 `todo`、`in_progress` 或 `blocked` 任务错误地标成 `fix`。这三种情况都应当走正常规划或修复 `blocked` 的流程。

## 示例

错误拆分：

- `TASK-002`：实现登录重试逻辑。
- `TASK-003`：验证 `TASK-002` 是否工作。

正确拆分：

- `TASK-002`：实现登录重试逻辑，补或改测试，并运行必要验证命令。

错误拆分：

- `TASK-004`：更新共享 API schema。
- `TASK-005`：更新依赖这个 schema 的后端处理逻辑。

正确拆分：

- `TASK-004`：把共享 API schema 和耦合的后端处理逻辑一起更新，并用检查覆盖联动行为。

正确修复示例：

- 用户指出交付的 `F-TASK-007-补充登录重试提示.yaml` 仍缺少所需恢复文案。
- 把它转换成 `fix-TASK-007-补充登录重试提示.yaml`，补上 `fix_request` 与 `fix_reason`，保留原完成记录，让 `after` 只重做这个受限任务。

## 工作流

1. 解析协作目录。
2. 读取用户当前需求。
3. 读取当前任务池，包括 `blocked` 和修复态任务。
4. 检查 `D:/3.3-19.01` 下相关仓库区域。
5. 设计候选任务，并逐个检查独立性、自证能力、依赖必要性、文件冲突风险，以及与现有未完成任务的冲突。
6. 对那些只是因为实现顺序、验证顺序或高冲突重叠而产生依赖的候选任务，优先通过合并或重画边界消除依赖。
7. 把剩余的未完成任务冲突写进各任务的 `conflicts_with`。
8. 决定本次需要多少任务。
9. 如果涉及挑战已完成任务的结果，先判断这是缺实现细节、问题定义错误，还是全新的相邻需求，再动任务池。
10. 在实际修改任务池前，再确认用户最新一条消息是在要求修改任务池，而不是只讨论策略或 skill 行为。
11. 如果当前允许写入，且用户确实要求修改任务池，就在 `<协作目录>/tasks` 中写新任务文件、按需原地修复 `blocked` 任务、或把 `done` 任务原地转为 `fix`。
12. 确保每个新建或修复后可执行的任务都以 `status: todo` 结束，而等待重做的已完成任务则以 `status: fix` 结束。

## 输出

写完任务后，用中文简短回复，必须包含：

- 使用了哪个协作目录。
- 新建了哪些任务文件，写完整文件名。
- 修复了哪些 `blocked` 任务。
- 把哪些已完成任务转换成了 `fix`。
- 哪些新建或修复后的任务带有 `conflicts_with`，以及它们分别冲突哪些未完成任务号。
- 是否仍然存在依赖。
- 现在 `after` 应该从哪个任务开始。
- 一句独立的 `本次关键决策：...`，说明这次规划里最核心的判断，尤其是在围绕用户真实目标重构任务时。
- 说明本次为什么被处理成新建任务、修复 `blocked`、把 `done` 转成 `fix`，或者只是做 dry-run 没有落文件。

不要输出长篇实现方案。任务池 YAML 才是唯一事实来源。
