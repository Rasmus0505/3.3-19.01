# Skill To MD

把这份文档当作“将 Codex skill 转成普通 Markdown”的操作说明。目标不是做摘要，而是产出一份其他 AI 仅靠这一个 `.md` 文件就能尽量完整执行原 skill 的独立说明。

## 目标

输出一个独立 `.md`，在行为、边界、流程、状态规则、输出约束上尽量与原始 skill 等价，使不具备 Codex skill 机制的其他 AI 也能直接按这份文档工作。

## 先确认的输入

在开始转换前，先明确：

- 源 skill 的名称或路径。
- 目标 Markdown 的保存路径。
- 目标文档是否保持原语言，还是要本地化。
- 用户要的是“严格等价导出”，还是“为非 Codex 运行时做可移植适配”。

如果用户没有指定新建 Codex skill 的放置路径，默认使用 `~/.codex/skills`。

## 需要读取的内容

按最小必要原则读取源内容：

1. 先读源 skill 的 `SKILL.md`。
2. 只有当 `references/`、`scripts/`、`agents/openai.yaml` 中存在会影响其他 AI 正确执行的规则时，才继续读这些文件。
3. 不要把无关的资源或整包引用都塞进上下文。

原始 skill 是唯一权威来源，其他派生文档只作参考。

## 必须保留的内容

除非用户明确要求修改，否则必须保留：

- 角色边界与职责划分。
- 固定路径、目录约定、文件命名规则。
- 状态机与状态流转顺序。
- 工作流步骤与停止条件。
- 硬性限制与安全规则。
- 必需的 schema、字段、文件名和字面量。
- 最终回复或输出格式要求。
- 会影响行为的工具要求或能力要求。
- 何时允许改文件、何时不能改文件的规则。

如果原 skill 里的示例有助于理解边界，也要保留或等价改写，不要因为“看起来像例子”就删掉关键判断依据。

## 必须适配的内容

为了让非 Codex AI 能直接用这份文档，要做这些适配：

- 去掉对 Codex 隐式 skill 触发机制的依赖。
- 只有在对目标读者有帮助时，才把 frontmatter 里的“什么时候触发”改写成正文说明。
- 如果原 skill 依赖外部引用文件中的关键规则，而其他 AI 不会自己继续打开这些文件，就必须把关键规则内联或明确补进导出文档。
- 对 Codex 专有能力，要改写成明确能力要求；但如果原始工具名本身属于操作契约，也要保留原名并说明其作用。
- 可以为可移植性补充解释，但不能借机放松原始约束。

## 转换规则

- 如果现有导出文档与源 skill 漂移明显，优先整份重写，不要做零碎补丁。
- 当用户要求高保真时，尽量保持与源 skill 接近的章节顺序。
- 把只在 Codex skill 体系里才成立的简写改成显式说明。
- 如果源 skill 链接的引用内容对正确执行是刚需，要么直接内联，要么把最小必要规则并入导出文档。
- 如果因为 Codex 专有能力导致无法做到完全等价，必须明确写出这个限制，而不是悄悄删掉行为。

## 质量检查

完成前检查以下几点：

- 另一个 AI 即使不打开原 skill 目录，也能只靠这份 Markdown 工作。
- 没有漏掉关键文件名模式、状态名、字段名、路径或硬限制。
- 导出文档没有弱化原始 skill 的规则。
- 导出文档内部自洽，不与源 skill 冲突。
- 如果做了本地化，语义仍保持原始操作要求，不因翻译而改变边界。

如果源 skill 和导出文档之间出现操作规则冲突，优先修正文档，而不是在回复里解释“差不多”。

## 工作流

1. 解析源 skill 和目标 Markdown 路径。
2. 读取源 `SKILL.md`。
3. 判断是否存在必须补充读取的引用文件。
4. 区分哪些规则可以保持简洁，哪些规则必须内联，才能让其他 AI 独立使用。
5. 起草或重写目标 Markdown，让它成为独立文档。
6. 如果用户还要求把这套方法沉淀成新的 Codex skill，先初始化 skill，再补全其 `SKILL.md`。
7. 如果新建或更新了 Codex skill，要验证该 skill 结构是否合法。
8. 最终回复里说明用了哪个源 skill、写到了哪个目标 Markdown，以及是否存在无法消除的可移植性 caveat。

## 输出

完成转换后，简短汇报：

- 使用了哪个源 skill。
- 写入或更新了哪个 Markdown 文件。
- 本次导出属于严格等价还是可移植适配。
- 是否存在必须保留的 Codex 专有行为说明。
