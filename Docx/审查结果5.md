## 一、汇总结论

- 本次整合了 4 份只读审查结果：主进程安全与架构、Preload 与渲染桥接、本地数据库与同步引擎、构建与打包流程。
- 四份审查共同指向同一个核心问题：当前桌面端的信任边界偏宽，远端页面、renderer、preload、主进程、本地数据库和本地 helper 之间缺少足够的分层与 capability 收口。
- 风险分布上，运行时权限边界问题最突出，其次是数据一致性闭环缺失，最后是构建供应链与安装包完整性不足。
- 本次仅整理和汇总审查结果，没有修改业务代码。

## 二、总体风险画像

| 维度 | 主要问题 | 风险等级 |
| --- | --- | --- |
| 页面到桌面能力边界 | 远端页面可获得高权限 preload / IPC 能力，缺少 sender / origin 校验 | critical |
| 渲染层到本地数据边界 | renderer 可直接读写认证缓存、本地数据库、同步元数据和冲突记录 | critical |
| 本地数据一致性 | 删除不记日志、同步成功不落 `synced`、进度同步键错配 | critical |
| 文件与 helper 访问 | 任意媒体路径可读、helper 代理边界随接口扩张而自动扩大 | high |
| 构建与分发链路 | 未签名、缺少资源完整性校验、依赖与下载源复现性不足 | high |

## 三、四份审查结果整合

### 1. 审查结果1：构建与打包流程

- 结论摘要：确认 6 项风险，集中在供应链完整性、安装包签名缺失、资源完整性、环境配置落盘、PyInstaller 动态导入验证不足和依赖复现性不足。
- 关键问题：
  - 第三方下载资源缺少独立 hash 或签名校验。
  - Windows 安装包未强制代码签名。
  - `asar` 关闭导致资源完整性保护不足。
  - URL 配置写盘时可能把敏感 query / userinfo 带进安装包。
  - 打包后缺少 helper 运行期冒烟验证。
  - 构建依赖受本机环境影响较大。

### 2. 审查结果2：Electron 主进程安全与架构

- 结论摘要：当前实现默认信任被加载页面，但主窗口既支持远端页面模式，又暴露了多项高权限桌面能力，主进程缺少发送方校验和导航限制。
- 关键问题：
  - 主窗口可导航到未知 origin，但 preload 仍保留高权限桥接。
  - 敏感 IPC 缺少 `senderFrame.url` / origin 校验。
  - `shell.openExternal` 对外链协议缺少白名单。
  - `read-local-media-file` 未绑定用户选择结果。
  - `safeStorage` 失败时回退为 `plaintext-base64`。
  - packaged 模式下全局劫持 `https`，架构耦合较重。

### 3. 审查结果3：本地数据库与同步引擎

- 结论摘要：确认 6 项数据库与同步风险，其中 2 项 critical、3 项 high、1 项 medium，已经影响本地与云端状态闭环。
- 关键问题：
  - `progress` 同步使用错误查询键，导致进度可能永远无法上云。
  - 删除课程不写 `DELETE` journal，远端拉取后会“复活”。
  - `sync_journals` 无法收敛，成功上云后也不 `markSynced`。
  - `sql.js` 到 IndexedDB 的全量快照持久化不是原子操作。
  - 坏快照 / WASM 初始化失败时无恢复路径。
  - 冲突检测与解决策略只覆盖 `courses`。

### 4. 审查结果4：Preload 与渲染桥接

- 结论摘要：当前 preload 向 renderer 暴露了过多高权限能力，尤其是认证凭据、数据库写接口、同步能力和本地文件读取入口。
- 关键问题：
  - `desktopRuntime.auth.restoreSession()` 可把 access / refresh token 交给渲染层。
  - `localDb` 与 `syncEngine` 暴露的是原始高权限接口，而不是窄业务接口。
  - `readLocalMediaFile` 允许按任意路径读取媒体文件。
  - `requestLocalHelper` 是通用代理，边界会随 helper 新接口自动扩张。

## 四、跨结果共性问题

| 共性主题 | 关联结果 | 说明 |
| --- | --- | --- |
| 信任边界过宽 | 结果2、结果4 | 远端页面、renderer、preload、main 之间默认互信，高权限能力缺少 origin / sender 验证。 |
| capability 未收口 | 结果2、结果4 | 暴露的是通用能力接口，不是最小权限业务接口，导致一处前端注入就能横向放大。 |
| 数据闭环缺失 | 结果3、结果4 | renderer 可操控同步与本地数据，而同步主流程本身还存在收敛缺陷，叠加后风险更高。 |
| 供应链与完整性不足 | 结果1、结果2 | 构建产物未签名、资源可篡改，运行时又依赖高权限桥接，使分发与运行风险叠加。 |

## 五、建议修复优先级

### P0

- 为主进程所有敏感 IPC 增加统一 sender / origin 校验。
- 阻断主窗口导航到未知 origin，或把远端页面与高权限桌面壳分离。
- 把 access / refresh token、auth cache、解密逻辑从 renderer 可见范围内移除。
- 收回 `localDb`、`syncEngine`、`requestLocalHelper` 的原始接口，只保留窄能力桥接。

### P1

- 修复 `progress` 同步主键错配、删除记录复活、`markSynced` 缺失等数据闭环问题。
- 将文件读取能力改为一次性句柄 / token 模式。
- 停止 `safeStorage` 失败时回退为可逆明文。

### P2

- 构建链路增加依赖锁定、下载校验、安装包签名和关键资源完整性校验。
- 为 sql.js 持久化引入坏快照恢复、双快照或增量日志机制。
- 为打包后的 helper 增加健康检查或最小启动测试。
