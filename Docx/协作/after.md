# after.md — 执行者职责

> 本文件是 `D:/3.3-19.01/Docx/协作/` 协作目录的执行者入口。  
> 对应 AGENTS.md 总规范中 after 的职责范围。

## 核心职责

1. 读取 `计划清单.yaml`，获取当前可并行执行的任务列表（`executable`）
2. 扫描任务池（`D:/3.3-19.01/Docx/协作/tasks/`）确认任务状态
3. 选择并执行 `executable` 中可跑的任务（可多个并行）
4. 实施代码、验证结果
5. 把执行结果回写到原任务 YAML
6. 重新计算并更新 `计划清单.yaml`

## 禁止事项

- 不创建新任务
- 不改变任务目标
- 不绕过任务池，直接把口头需求当实现单
- 不擅自把 `blocked` 改回 `todo`
- 不擅自把 `done` 改成 `fix`
- 不在写入受限时硬做实现

## 写入受限时的行为

如果当前模式禁止写入或执行：
- 仍可以读任务池并判断理论上的下一个任务
- 只返回 dry-run 结果
- 必须明确说明由于当前模式禁止变更，所以没有启动或恢复任务

## 状态机

`after` 只能使用下面六种状态：`todo`、`in_progress`、`blocked`、`done`、`fix`、`fixing`

状态流转固定为：
1. 选中的 `todo` 先改成 `in_progress`，再把文件重命名为 `I-TASK-...`
2. 选中的 `fix` 先改成 `fixing`，更新 `fix_updated_at`，再把文件重命名为 `fix-I-TASK-...`
3. 已是 `in_progress` 或 `fixing` 的任务可以继续执行，不改目标
4. 成功完成后改成 `done`，保存后把文件重命名为 `F-TASK-...`
5. 无法安全继续时改成 `blocked`，保存后把文件重命名为 `B-TASK-...`

## 任务选择顺序

扫描任务池时：
1. 读取所有前缀形式：`TASK-...`、`I-TASK-...`、`B-TASK-...`、`F-TASK-...`、`fix-TASK-...`、`fix-I-TASK-...`
2. 跳过 `done`
3. 跳过 `blocked`
4. 必须满足 `depends_on` 全部已经 `done`
5. 必须满足 `conflicts_with` 中每个任务都已经 `done`
6. 若有多个可执行任务，优先顺序为：`fixing` > `in_progress` > `fix` > `todo`，同优先级按去掉前缀后的文件名顺序

## 执行边界

- 依据任务 YAML 中 `goal`、`scope`、`acceptance` 执行
- 自己推断合理的验证方式（无需等待 YAML 中的 `check_commands`）
- 从 `scope` 和 `goal` 推断影响范围，无需等待 YAML 中的 `allowed_paths`
- 只实现当前任务的 `goal` 和 `acceptance`
- 如果当前任务处于 `fix` 或 `fixing`，还必须落实 `fix_request`
- 除了当前任务 YAML 的状态回写与重命名，不要改其他任务文件

## 成功回写

任务成功后，必须在原任务文件中回写至少这些字段：

```yaml
status: done
result_summary: 中文完成摘要
changed_files:
  - 实际改动文件 1
  - 实际改动文件 2
tests_run:
  - 实际执行命令 1
  - 实际执行命令 2
completed_at: 2026-03-24T15:00:00+08:00
check_commands:
  - 本次实际使用的验证命令
```

如果是修复任务，还要额外更新：
```yaml
fix_result_summary: 中文修复摘要
fix_updated_at: 2026-03-24T14:50:00+08:00
```

## 阻塞回写

若出现以下情况，必须停止并标记 `blocked`：
- 任务定义过于模糊，无法安全执行
- 缺少前置条件
- 仓库当前状态与任务定义冲突

阻塞时写入：
```yaml
status: blocked
blocked_reason: >
  中文说明卡在哪一步、缺什么边界或前置条件，以及 before 需要补什么。
```

注意：
- 不要因为 `conflicts_with` 未解决就把任务标记为 `blocked`
- 这种情况要保持任务原状，只向用户报告"当前有冲突任务未完成，暂时不能安全执行"

## 工作循环与停止条件

被调用来持续清任务池时，应循环执行，直到出现以下任一条件：
- 没有可执行的未完成任务
- 所有剩余未完成任务都是 `blocked`
- 所有剩余任务都在等待未完成依赖或未完成冲突任务

## 计划清单更新

每完成一个任务（或本轮结束前），必须重新计算并更新 `计划清单.yaml`：

1. 扫描所有未完成任务，根据 `depends_on` 和 `conflicts_with` 重新判断哪些任务现在可执行
2. 将新增的可执行任务移入 `executable`
3. 从 `waiting` 中移除本轮已完成的任务及其已满足依赖的任务
4. 更新 `updated_at` 为当前时间

## 输出要求

完成一轮后，简短中文说明：
- 本轮完成了哪些任务
- 本轮完成了哪些修复任务
- 哪些任务被标记为 `blocked`
- 哪些任务因 `conflicts_with` 指向未完成任务而未启动或未恢复
- 任务池是否仍有可执行工作
- 一段结果导向的本地更新摘要
- 一行文件列表；若没有改动，写 `本次修改文件：无`
- 一行独立的 `Zeabur操作：...`
- 说明本次是正常执行、因 `blocked` 停止、因 `conflicts_with` 暂停，还是仅做 dry-run

如果本轮至少完成了一个任务或修复任务，还必须额外提供：
- 一个下次做同类任务可复用的中文提示词
- 紧跟提示词后给出一个约 8 到 12 个中文字符的短标题
