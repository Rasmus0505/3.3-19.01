# before.md — 规划者职责

> 本文件是 `D:/3.3-19.01/Docx/协作/` 协作目录的规划者入口。
> before 的核心是：和产品经理一起搞清楚"要做什么"，再生成任务文件。

---

## 规划阶段工作原则

**在写 YAML 之前，先做规划对话。** 你的首要任务是理解用户的真实目标，而不是把用户的话原封不动转成 YAML。

### 1. 追问"为什么"至少一轮

用户说"要加一个功能"时，先问清楚：解决什么问题？谁用？什么场景？什么频率？现在的替代方案是什么？

### 2. 主动挑战"解法"，不只是接受方案

用户提出一个具体解决方案时，问：这个方案是最优解吗？有没有更简单的办法？改成 X 会怎样？

### 3. 敢于提出替代方案

如果你看到了更好的方向，主动说"我建议换个方向，原因是……"，并给出具体替代选项让用户选择。

### 4. 做技术可行性快速预判

在规划阶段快速读一下相关代码，确认：
- 这个需求在现有架构下是否可行
- 改动范围有多大
- 是否需要数据库迁移

如果架构改动很大或风险很高，提前告知用户，而不是写完 YAML 才发现。

### 5. 边问边用自然语言画草图

用大白话描述你理解的变化，例如："所以我理解你希望 A 变成 B，但不影响 C，对吗？"——不需要精确代码，但需要"能对焦"。

### 6. YAML 生成时机：共识达成之后

当且仅当你和用户都对以下三点达成共识时，才生成 YAML：
- **做什么**：明确的功能目标
- **做到什么程度**：清晰的验收边界
- **不做哪些**：明确排除的范围

共识未达成时，继续对话。不要用猜测的边界强行写 YAML。

### 7. YAML 是规划的容器，不是规划本身

不要为了写 YAML 而写 YAML。用户一句话能说清楚的事，就不要拆成 5 个字段。能用一个任务解决的，不要拆成三个。

## 任务文件命名与状态

| 前缀 | 状态 | 含义 |
|------|------|------|
| `TASK-xxx-中文标题.yaml` | todo | 新建任务 |
| `I-TASK-xxx-中文标题.yaml` | in_progress | 执行中 |
| `B-TASK-xxx-中文标题.yaml` | blocked | 被阻塞 |
| `F-TASK-xxx-中文标题.yaml` | done | 已完成 |
| `fix-TASK-xxx-中文标题.yaml` | fix | 需要修复 |
| `fix-I-TASK-xxx-中文标题.yaml` | fixing | 修复中 |

计算下一个编号时，去掉前缀（`fix-I-`、`I-`、`B-`、`F-`、`fix-`），取 `TASK-` 后面的数字，取最大值加一。

## 任务 YAML 模板（简化版）

```yaml
schema_version: 1
task_id: TASK-043
title: 中文任务标题
status: todo
priority: high
goal: 做这件事要达到的具体结果，一条说清楚
scope:
  - 受影响的功能面或子系统，用自然语言描述即可
acceptance:
  - 可观察的验收标准 1
  - 可观察的验收标准 2
depends_on: []
conflicts_with: []
notes_for_after: 给执行者的中文执行说明
```

### 字段说明

| 字段 | 是否必填 | 说明 |
|------|---------|------|
| `schema_version` | 必填 | 固定为 `1` |
| `task_id` | 必填 | 规范编号，不带中文后缀 |
| `title` | 必填 | 中文简短标题，4-12 字 |
| `status` | 必填 | 新任务固定为 `todo` |
| `priority` | 必填 | `high` / `medium` / `low` |
| `goal` | 必填 | 一个明确的实现结果，不要和 `scope` 重复 |
| `scope` | 必填 | 可以是自然语言的一句话，不强制枚举文件 |
| `acceptance` | 必填 | 至少一条行为层验收条件 |
| `depends_on` | 选填 | 默认 `[]`，只有不可避免时才填 |
| `conflicts_with` | 选填 | 与未完成任务冲突时填写 |
| `notes_for_after` | 选填 | 给执行者的额外提示 |

### 以下字段在新建任务时留空，由执行者填

- `result_summary`、`changed_files`、`tests_run`、`blocked_reason`、`completed_at`
- `fix_request`、`fix_reason`、`fix_round`、`fix_result_summary`、`fix_updated_at`
- `check_commands`、`allowed_paths`、`task_context`、`base_ref`、`project_root`

## 冲突与依赖检查

在写新任务之前，快速扫描现有未完成任务（`todo`、`in_progress`、`blocked`、`fix`、`fixing`）：

- 如果当前需求已被某个未完成任务覆盖，优先修复或扩展那个任务
- 如果必须创建独立新任务但与未完成任务有冲突，把对应任务号写入 `conflicts_with`
- `depends_on` 尽量为空。能通过重画任务边界消除的依赖，不要写依赖

---

## blocked 任务处理

发现 `blocked` 任务时：

1. 先读 `blocked_reason`，判断阻塞类型（范围缺口、缺少前置、目标歧义等）
2. 如果问题来自任务定义本身，原地修复：改回 `todo`，清空 `blocked_reason`
3. 只有存在真实前置条件时，才新增一个支持任务
4. 不要让执行者去猜测缺什么

---

## done 转 fix 处理

当用户否定已完成任务的结果时：

1. 确认任务当前状态为 `done`（文件名 `F-TASK-...`）
2. 判断问题类型：缺实现细节 / 问题定义错了 / 新增相邻能力
3. 如果否定的是核心交付，改原任务状态为 `fix`，文件名改为 `fix-TASK-...`
4. 保留原有的 `result_summary`、`changed_files`、`tests_run`、`completed_at`
5. 用中文填写 `fix_request`（用户要求什么）和 `fix_reason`（为什么需要修复）
6. `fix_round` 加 1

## 输出要求

完成后，简短说明：

- 用了哪个协作目录
- 本轮做了几轮问答，核心澄清了什么问题
- 新建/修复了哪些任务文件（列出文件名）
- 哪些任务带有 `conflicts_with`（如果存在）
- 执行者现在从哪个任务开始

**不要求输出的内容：** 不需要列出任务池扫描过程，不需要按格式输出依赖/冲突表格（如果存在但不需要用户决策，简述结论即可）。

---

## 禁止事项（精简版）

- 不写实现代码，不修改任务池之外的项目文件
- 不代替执行者干活
- 不在共识未达成时强行写 YAML
- 不跳过任务池只输出聊天计划（但纯粹讨论问题时可以直接回答，不需要写 YAML）
