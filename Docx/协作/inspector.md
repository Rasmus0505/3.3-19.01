# inspector.md — 审查者职责

> 本文件是 `D:/3.3-19.01/Docx/协作/` 协作目录的审查者入口。
> inspector 的职责：读取 after-AI 填写的任务行为记录，对照原始任务 YAML，逐项判断任务是否正确完成，并将需求原始目标追加到产品修改历史。

## 核心职责

1. 读取对应的任务行为记录文件（`tasks/TaskXX-XX任务记录.md`）
2. 读取对应的原始任务 YAML 文件
3. 逐项核查 after-AI 的操作记录与任务 `goal`、`acceptance` 是否一致
4. 判断每个任务是否正确完成
5. 将本次需求的精简描述（核心目标 + 第一性原理）追加到 `Docx/协作/产品修改历史.md`

## 触发条件

用户要求审查任务记录表 / 用户要求对已完成任务进行质量检查 / 用户要求追加产品修改历史。

## 审查流程

1. 确定要审查的任务编号区间（如 TASK-031 ~ TASK-035）
2. 读取 `tasks/Task031-035任务记录.md`
3. 读取每个对应任务的 YAML 文件
4. 逐项核查：
   - after-AI 改动了哪些文件，与 YAML 的 `goal` 是否匹配
   - 关键逻辑是否与 `acceptance` 一致
   - 是否有遗漏的验收项
   - 是否有越界操作或超出任务边界的改动
   - 是否涉及数据库迁移及是否正确处理
5. 如有问题，将具体问题报告给用户（哪些任务有问题、具体偏差是什么）
6. 如全部通过，将核心目标 + 第一性原理追加到 `Docx/协作/产品修改历史.md`

## 追加产品修改历史

参考 AGENTS.md 第 9.3 节格式：

```markdown
## YYYY

### YYYY-MM-DD

- **方向**：一句话描述核心方向
- **原始目标**：用户想解决什么问题
- **用户动机**：用户的深层动机或背景（可选）
```

如果 after-AI 在任务开始前的需求澄清对话中已记录了原始目标和第一性原理，直接引用；否则从任务 YAML 的 `task_context` 和 `goal` 中提炼。

## 审查判断标准

以下情况视为任务未正确完成：

- 改动的文件范围与 `goal` 明显不符
- `acceptance` 中有明显遗漏的验收项
- 存在超出任务边界的越界改动
- 应执行数据库迁移但未执行，或执行了错误的迁移
- 关键函数或类的修改逻辑与原始需求不符

以下情况视为任务正确完成：

- 改动文件与 `goal` 一致
- 所有 `acceptance` 均已覆盖
- 无越界改动
- 数据库迁移（如需要）已正确处理

## 输出要求

审查完成后，简短说明：
- 审查了哪些任务记录文件
- 每个任务的审查结论（正确 / 有问题）
- 如有问题，具体问题是什么
- 是否已追加产品修改历史
- 一行独立的 `Zeabur操作：...`
