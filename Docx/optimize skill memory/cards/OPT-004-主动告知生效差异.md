# 主动告知生效差异

## Symptoms

- `after` 的原始输出要求偏向任务完成摘要，没有强制把“这次改动会改变以后怎么启动、验证、打包或部署”单独告诉用户。
- 当改动需要重新打包、重装、重启、迁移、改环境变量或改验证方式后才会真正生效时，用户容易沿用旧路径操作，误以为本次修改没有生效。
- 原规则虽然保留了 `Zeabur操作：...`，但它只覆盖平台动作，无法承载更广的用户感知行为差异。

## Root Cause

- `after` 的输出契约缺少专门面向用户后续操作的固定区块，导致执行器可以把高影响差异埋进普通总结，甚至完全省略。
- 技能没有要求执行器在完成实现后主动复盘“本次是否改变了用户的使用方式或生效条件”。
- 技能对不完全确定但高度疑似的行为变化没有“宁可提醒”的策略，容易因为证据不满 100% 而沉默。

## Optimization Principle

- 执行器的价值不只是汇报“代码改了什么”，还要明确告知“用户接下来要怎么做，结果才会按预期生效”。
- 对用户后续使用方式的变化要独立输出，不能让用户自己从摘要中猜。
- 遇到高概率会影响生效或验证路径的变化，应该优先提醒，再说明不确定性，而不是等完全确认后才说。

## Suggested Skill Changes

- `SKILL.md` 输出契约：
  - 强制新增三行独立输出：`本次行为差异：...`、`生效条件：...`、`你需要做的事：...`。
  - 即使没有行为差异，也要显式写出否定结论，避免用户自行猜测。
  - 对高度疑似但未完全确认的差异，要求在这三行中直接写明 `可能`、`请确认`、`需要按以下方式验证`。
- `SKILL.md` 执行规则：
  - 在成功实现或修复后，新增一步显式复盘，检查启动方式、验证方式、打包/安装/部署路径、环境变量、迁移、重启/重装要求、默认行为等是否发生变化。
  - 明确这些变化属于必须主动提醒的“用户感知行为差异”。
- 导出文档：
  - 同步更新独立的 `after.md`，保证外部 AI 读取 Markdown 版本时也会执行同样的告知规则，而不是只有源 skill 更新。

## Applicability

- 本次改动会改变用户之后如何启动、安装、验证、部署、配置、更新或使用功能时。
- 本次改动需要额外动作后才生效，例如重打包、重装、重启、迁移、重部署、改环境变量或清缓存时。
- 本次改动让旧验证方式不再可靠，需要用户改用新的入口、命令或安装包时。

## Non Applicability

- 纯内部重构且不改变用户后续任何操作方式，也不影响生效条件或验证路径时。
- 当前回复不是在汇报一次执行结果，而只是讨论策略、复盘技能或规划优化方案时。
