# 定向并发失真
## Symptoms

- 用户输入带数字参数的 `$after 18`，执行器没有先解释该数字含义，而是按默认队列顺序先执行 `TASK-016`。
- 执行过程中任务池被并发改写，`TASK-016`、`TASK-019` 在主线程观察窗口内被外部改成 `done`，执行器只能临时停在 `TASK-018` 前避免覆盖。
- 最终结果虽然安全收尾，但依赖主线程临场判断，不是技能本身提供的稳定分支。

## Root Cause

- `after` 的触发与边界规则只定义了“执行任务池”，没有定义 `$after 18` 这类带目标参数时是“定向执行某个 task_id”还是“继续全队列”。
- `after` 的工作流默认单执行者视角，缺少“每次选题前、写回前、继续下一题前强制重扫任务池”的并发保护步骤。
- 技能没有要求当当前任务文件被外部重命名或状态变更时，按 `task_id` 重新定位并以磁盘状态为准。

## Optimization Principle

- 当用户输入比技能默认入口更具体时，必须优先解释该具体性，不能静默退回泛化流程。
- 共享任务池是并发环境，执行器每次关键转移都必须重新以磁盘状态为真源，而不是延续内存中的旧选择。
- 技能里需要把“无法确定用户是点名任务还是要跑全队列”视为边界问题，不应靠运行中补救。

## Suggested Skill Changes

- `SKILL.md` 触发/边界：
  - 明确新增“`$after <数字>`、`$after TASK-018`、`继续 018`”的定向执行分支。
  - 要求先将数字解析为目标 `task_id`；若无法唯一匹配，直接停下并向用户澄清，而不是退回全队列执行。
- `SKILL.md` 工作流：
  - 在“选出下一任务”前、`in_progress/done/blocked` 写回前、每完成一题继续下一题前，强制重新扫描整个任务池。
  - 若原任务文件路径不存在、文件名前缀变化、或磁盘上的 `status` 已被外部改写，必须按 `task_id` 重新定位并采用磁盘状态。
  - 若检测到并发执行者已经完成当前任务或完成了更高优先级任务，当前执行器必须放弃旧内存计划并重新排序，而不是顺延旧队列。
- `SKILL.md` 输出：
  - 新增一条输出要求：当本轮因为并发改写而重选任务或提前停下时，要在最终回复中明确说明“并发状态变更导致重扫/停下”。
- `references/`：
  - 增加一个短参考，专门列出“定向执行输入样式”和“并发任务池重扫时机”的例子，减少执行器自由解释空间。

## Applicability

- 用户用 `$after` 并追加数字、task id、文件名片段，明显不是单纯“跑完整队列”时。
- 任务池可能被多个 AI、人类或自动流程同时修改时。
- 任务文件在执行中会被重命名、状态切换或被其他执行者写回时。

## Non Applicability

- 用户只输入裸 `$after`，且环境明确是单执行者、不会有并发改写时。
- 当前会话只是讨论 `after` 策略或复盘结果，没有要求真的消费任务池时。
