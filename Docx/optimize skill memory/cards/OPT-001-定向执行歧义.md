# 定向执行歧义

## Symptoms

- 用户输入 `$after 19`，显然想点名处理任务 19。
- `after` 技能正文只定义了“扫描任务池并选择下一个可执行任务”，没有定义带任务号参数时的分支。
- 执行器因此先额外查 `after.md` 和任务池状态，再临时决定是否按用户点名覆盖默认排序。
- 这会让“继续执行任务池”和“只执行指定任务”之间的优先级变成现场推断，而不是技能内规则。

## Root Cause

- `after` 的触发描述把所有 `$after` 调用都归为同一类，但 workflow 没有区分无参数续跑与带参数定向执行。
- 技能缺少“如何解析 19 / TASK-019 / 文件名前缀变体”的明确规则。
- 技能也没有规定：当用户点名任务与默认排序冲突时，是优先尊重点名、拒绝执行，还是退回默认队列。

## Optimization Principle

- 任务池执行技能必须先判定“用户是在续跑整个队列，还是在请求定向执行某个任务”。
- 对带任务号的执行请求，技能必须给出稳定、可复用的选择规则，不能依赖主线程临时解释。
- 当定向请求不满足依赖、冲突或状态约束时，应明确拒绝并说明原因，而不是静默改跑别的任务。

## Suggested Skill Changes

- `description`:
  - 明确区分两种入口：`$after` 表示按任务池规则继续执行；`$after <task-id>` 表示尝试定向执行指定任务。
- `SKILL.md` workflow:
  - 在“Find the next executable unfinished task”之前新增“解析用户是否点名任务”的步骤。
  - 支持将 `19`、`TASK-019`、`I-TASK-019...yaml` 统一归一到同一任务 id。
  - 规定定向执行时只检查该任务自身的 `depends_on`、`conflicts_with`、`status` 与 `allowed_paths`，不要再退回全局排序。
  - 规定若指定任务不可安全执行，应保持任务池不变并直接向用户报告阻塞原因。
- `SKILL.md` task selection:
  - 把现有排序规则明确限定为“仅用于未指定任务号的 `$after` 调用”。
- `SKILL.md` output:
  - 当是定向执行请求时，结果中补一句“本轮为定向执行 TASK-xxx，未继续消费后续可执行任务”。
- `agents/openai.yaml`:
  - 将默认提示改成既覆盖“继续执行”也覆盖“定向执行”两种模式，避免代理天然只按全局队列理解 `$after`。

## Applicability

- 用户用 `$after 19`、`$after TASK-019`、`$after fix-TASK-003` 这类形式点名任务时。
- 任务池存在多个可执行任务，但用户只想推进某一个任务时。

## Non Applicability

- 用户只输入 `$after`，明确是让执行器按默认任务池顺序继续消费队列时。
- 用户是在讨论任务池策略、技能行为或让执行器先分析是否应该执行时。
