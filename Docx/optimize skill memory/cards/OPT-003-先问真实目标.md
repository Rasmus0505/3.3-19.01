# 先问真实目标

## Symptoms

- 用户给出一个方案或半成品描述后，技能过早进入建任务流程，没有持续追问现状问题、真实目标和边界。
- 生成的任务更像动作清单，缺少“为什么会有这个任务、当前哪里不对、完成后要纠正成什么样”的起因说明。
- 技能容易让用户补技术实现细节，而不是先由技能自己从仓库里补技术事实。

## Root Cause

- 技能虽然有“围绕真实目标”的原则，但没有把它写成强制门槛，导致执行时容易退化为顺着用户方案拆任务。
- YAML 输出契约缺少专门承载任务起因的字段，背景信息被挤压到 `goal`、`notes_for_after` 或聊天回复里，无法稳定传递给 `after`。
- 需求澄清与技术探索的责任边界不够清晰，容易把本该由技能自行确认的技术问题反抛给用户。

## Optimization Principle

- 在 tasking 前，先把“问题定义”问清，再做“方案定义”。
- 对用户的第一句话保持怀疑，把它视为线索，不视为完整规格。
- 追问重点放在业务目的、痛点、目标结果和边界，技术实现优先通过仓库检查获得。
- 只有当现状问题、目标结果和任务边界都明确后，才允许落任务。

## Suggested Skill Changes

- 在 `SKILL.md` 的角色和工作流中加入强制性的意图澄清回合，要求默认进行交互式追问。
- 增加明确的建任务门槛：真实问题、目标结果、任务边界三者任一不清，都不能写占位任务。
- 在任务 YAML 中新增 `task_context` 必填字段，固定写任务起因、现状缺口和目标状态。
- 明确禁止把代码结构、文件路径和实现细节决策优先抛给用户；除非那是仓库无法回答的真实产品取舍。
- 在 done 转 fix 和 blocked 修复流程中，同步要求重写 `task_context`，确保返工任务也从真实问题出发。

## Applicability

- 用户表达的是方案、操作建议或实现动作，但未明确业务目的时。
- 用户自己也不确定最佳方案，希望技能从第一性原理反推任务时。
- 任务将被其他执行代理接手，必须把“为什么做”写进任务文件时。

## Non Applicability

- 用户目标、问题和边界已经非常明确，且仓库事实也已足够支撑 tasking 时，不需要为了形式而无限追问。
- 纯技术讨论、非 task-pool 写入请求，或上层模式明确禁止变更任务池时，这张卡不改变原有安全边界。
