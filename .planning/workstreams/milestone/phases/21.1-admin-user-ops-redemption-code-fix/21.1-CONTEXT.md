# Phase 21.1: 后台管理用户运营优化 - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

优化管理台"用户运营"和"兑换码"两个页面的信息展示与操作：
1. 用户列表增加"用户名"和"注册时间"列
2. 兑换码操作从"启用/停用/废弃"精简为"删除/废弃"
3. 每个兑换码右侧增加复制按钮，复制明文兑换码（不脱敏）
4. 兑换码状态列合并为单一中文状态

本阶段只涉及前端展示调整和后端操作行为变更，不新增管理台页面，不改动兑换码生成逻辑。

</domain>

<decisions>
## Implementation Decisions

### 用户列表增强
- **D-01:** 用户列表表格在"邮箱"列后增加"用户名"列，显示 `User.username` 字段
- **D-02:** 用户列表同时增加"注册时间"列，显示 `User.created_at`（北京时间格式化）
- **D-03:** 用户名列允许空值显示（`username` 字段 `nullable=False`，实际应始终有值，但展示层做防御）

### 兑换码删除行为
- **D-04:** 兑换码"删除"为硬删除——数据库记录彻底移除，兑换码立即失效，不可恢复
- **D-05:** 批量操作"批量停用"按钮保留（对应 `bulk-disable` 接口），因为批量停用和单个废弃语义不同

### 兑换码废弃行为
- **D-06:** 后端废弃接口（`abandon`）执行时，自动遍历该兑换码的兑换记录，将对应金额从用户钱包扣除，生成负向流水（`WalletLedgerEntry`，类型为"废弃扣回"）
- **D-07:** 废弃操作仅针对已兑换（`effective_status=redeemed`）的兑换码，未兑换的兑换码废弃后直接删除（不产生扣款）
- **D-08:** 后端废弃扣款逻辑需加事务保护，确保扣款流水记录和钱包余额更新在同一事务中完成

### 兑换码复制按钮
- **D-09:** 复制按钮位于每行操作列最右侧（启用/停用/废弃按钮右侧），独立按钮
- **D-10:** 点击复制按钮直接复制完整明文兑换码（`code` 原始字段），不脱敏，不带格式化
- **D-11:** 复制成功后显示 toast 提示"兑换码已复制"

### 状态栏合并
- **D-12:** 兑换码列表删除"有效状态"列（`effective_status`），只保留"状态"列，改为中文单一状态
- **D-13:** 合并后的中文状态映射规则：
  - `unredeemed` + `enabled` → **正常**
  - `unredeemed` + `disabled` → **停用**
  - `redeemed` + 任意 → **已兑换**
  - `expired` + 任意 → **失效**
  - `abandoned` + 任意 → **废弃**
- **D-14:** 状态 Badge 使用统一的 `Badge` 组件，中文文字展示

### Claude's Discretion
- 前端用户名列的列宽和排序优先级
- 状态 Badge 的颜色风格（warning/danger 等配色）
- 删除确认对话框的具体文案
- 废弃扣款后用户钱包余额可能为负时的展示方式

</decisions>
