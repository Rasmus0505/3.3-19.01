# Phase 2.1: Admin Bottle 1.0 Settings & Billing Cleanup - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Repair the incomplete Phase 2.1 admin cleanup so the management console no longer has a standalone `/admin/models` surface or any model-parameter configuration workflow, while Bottle 1.0 becomes a normal billable model inside an existing admin workspace. This phase is about removing the old model-configuration control plane and aligning Bottle 1.0 billing end-to-end; it does not add learner-facing generation UX or expand Phase 3/4 scope.

</domain>

<decisions>
## Implementation Decisions

### Admin information architecture
- **D-01:** Remove the standalone `/admin/models` route and the "模型配置" navigation entry entirely.
- **D-02:** Model billing belongs inside an existing admin workspace rather than a dedicated model-configuration page.
- **D-03:** The surviving admin placement may reuse an existing workspace such as the user/business billing workspaces, but it must not recreate a separate model-config area under a different name.

### Model-parameter configuration removal
- **D-04:** Admin should not provide any model-parameter configuration workflow after this phase, not just for Bottle 1.0.
- **D-05:** Delete the old Bottle 1.0 dedicated admin configuration flow end-to-end instead of hiding it behind legacy or placeholder UI.
- **D-06:** Remove the associated frontend surfaces, backend endpoints, and persistence tables for the deprecated model-configuration control path.
- **D-07:** Default-strategy / legacy-model admin surfaces that still act as model-configuration controls should be removed rather than preserved in a slimmed-down `/admin/models` replacement.

### Bottle 1.0 billing alignment
- **D-08:** Bottle 1.0 must appear as a normal model row in the surviving "计费与启停" editor, with the same enable/disable and price-edit flow used by other billable models.
- **D-09:** Actual Bottle 1.0 billing lookup and deduction must use the same canonical `model_name` that the admin billing row edits.
- **D-10:** It is not enough for Bottle 1.0 to merely render in the table; the admin row, public/admin rate APIs, and runtime billing path must all resolve to the same model key.

### Completion guardrails
- **D-11:** This phase is not complete if `/admin/models` or any equivalent standalone model-config entry still exists.
- **D-12:** This phase is not complete if Bottle 1.0 is still configurable through a dedicated backend settings contract, even if the UI was removed.
- **D-13:** This phase is not complete if Bottle 1.0 is missing from the real admin billing list or if runtime billing still bypasses that row.
- **D-14:** This phase does not add new learner-facing Bottle 1.0 UX, link import, or Phase 3 lesson-normalization work.

### the agent's Discretion
- Which existing admin workspace becomes the final home for the billing editor, as long as `/admin/models` and the "模型配置" nav are removed
- The exact migration strategy for removing deprecated model-config tables safely
- The non-admin source of any remaining runtime defaults after subtitle/model-config admin controls are removed

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `frontend/src/features/admin-rates/AdminRatesTab.jsx`: already supports per-model enable/disable and price editing through `/api/admin/billing-rates`.
- `frontend/src/features/admin-workspaces/AdminUsersWorkspace.jsx`: already exposes "计费配置" inside an existing workspace, which fits the new IA decision.
- `frontend/src/features/admin-workspaces/AdminBusinessWorkspace.jsx`: already has a reusable section/panel model for embedding the billing editor in an existing admin workspace.
- `app/services/billing.py`: already includes `faster-whisper-medium` in canonical billing constants and default seed data, so billing alignment should extend the real path rather than invent a new one.

### Established Patterns
- Admin navigation is centrally defined in `frontend/src/shared/lib/adminSearchParams.js`, so removing "模型配置" requires routing and nav cleanup together.
- Billing is keyed by canonical `model_name` and should stay on that path all the way from admin editing to runtime deduction.
- Existing workspaces already consolidate related admin functions, so billing does not need its own dedicated route.

### Integration Points
- Remove `/admin/models`, its redirects, and the nav label from `frontend/src/AdminApp.jsx` and `frontend/src/shared/lib/adminSearchParams.js`.
- Re-home billing access inside an existing workspace without recreating default-strategy or legacy-model sections.
- Delete deprecated subtitle / legacy model-config UI and the backend endpoints/tables that supported those controls.
- Verify Bottle 1.0 runtime billing resolves through the same `model_name` row exposed in `/api/admin/billing-rates` and the public billing contract.

</code_context>

<specifics>
## Specific Ideas

- The previous 2.1 direction was conceptually right but operationally incomplete: `/admin/models` still exists, `Legacy ASR` still renders, and Bottle 1.0 is still not visible as the expected billing row.
- "模型配置" should disappear as an admin concept; the only surviving admin control in this area is billing/enablement inside an existing workspace.
- Deleting only the frontend is explicitly not enough; old interfaces and related tables must also be removed so no hidden control plane remains.
- Acceptance must prove that editing Bottle 1.0 billing affects the real runtime billing path through the same canonical `model_name`.

</specifics>
