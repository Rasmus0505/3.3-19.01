# Phase 07.1: Memo 模式复刻与桌面媒体工作流产品化 - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

在不改变既有产品边界的前提下，定义 Bottle 如何复制 Memo 的桌面媒体工作流模式：把公开链接导入、本地下载、本地媒体准备、课程生成与后续学习闭环收口成正式的桌面端产品能力，而不是继续停留在“可用但分散的 helper 功能”。本阶段明确复制深度、用户可见控制面、对外支持承诺和元数据利用程度，不把网页端变成 Bottle 1.0 可执行入口，也不把 Bottle 扩展成受限内容绕过工具。

</domain>

<decisions>
## Implementation Decisions

### 能力深度边界
- **D-01:** Bottle 在本阶段只复制 Memo 的公开链接工作流，不复制 `cookies / 账号登录 / 会员内容 / 受限内容导入` 能力。
- **D-02:** 桌面端链接导入能力边界固定为：常见公开视频页链接 + 常见公开直链；当内容命中登录态、会员限制、私有视频或平台受限场景时，必须明确失败，而不是进入绕过能力建设。
- **D-03:** Bottle 复制 Memo 的重点应放在“公开链接导入 -> 本地下载/转码 -> 课程生成 -> 历史/学习”的完整产品链路，而不是优先追求受限站点的下载成功率。

### 用户可见控制面
- **D-04:** 保持低技术感知路线。默认用户界面只暴露链接输入、课程标题、进度、取消、失败提示与后续生成动作，不新增明显可见的媒体工具面板。
- **D-05:** 本阶段不向普通用户暴露 `yt-dlp` 更新、下载源选择、cookie 管理、登录态管理或类似 Memo 的高级控制项。
- **D-06:** 如果后续需要调试或排障能力，也应优先走内部诊断或运营排障入口，而不是把媒体工具控制项直接前置到用户导入流程。

### 对外支持承诺
- **D-07:** 产品对外承诺采用“列举少量明确示例平台”的方式，而不是宣称支持所有 `yt-dlp` 可解析站点。
- **D-08:** 对外文案应表达为“支持常见公开视频链接”，并用 3-4 个常见示例帮助建立预期，例如 `YouTube`、`B站`、`常见播客页面`、`公开视频直链`。
- **D-09:** 这些平台示例用于设定用户预期，不构成需要长期维护的全量兼容平台清单；实现和测试应围绕“常见公开场景”建立稳定性，而不是追求平台广度。

### 元数据利用程度
- **D-10:** 本阶段链接导入后只稳定利用解析到的标题，作为课程默认标题来源；不把封面、频道、作者、来源平台等元数据升级成新的用户主界面资产。
- **D-11:** 导入成功后的产品目标是“课程生成更顺滑”，而不是“形成完整媒体卡片系统”，因此元数据利用保持最小化，避免本阶段滑向媒体资产产品化。

### the agent's Discretion
- 公开链接支持示例在最终文案中的排列顺序和措辞，只要维持“常见公开场景”口径即可。
- 标题写回课程的具体时机和失败回退策略，只要满足“标题优先于通用占位名”即可。
- 为内部调试保留哪些非用户可见运行信息，只要不破坏低技术感知原则即可。

</decisions>

## Existing Code Insights

### Reusable Assets
- `frontend/src/features/upload/UploadPanel.jsx`: 已有 `sanitizeDesktopLinkInput()`、`submitDesktopLinkImport()`、`pollDesktopLinkImportTask()`、`openSnapAnyFallback()` 和标题联动逻辑，说明前端工作流不是从零开始
- `app/api/routers/desktop_asr.py`: 已有 `_probe_ytdlp_metadata()`、`_download_media_with_ytdlp()`、`download_public_media()`、`create_url_import_task()`，说明后端 helper API 已具备真实下载链路
- `desktop-client/electron/main.mjs`: 已有 `desktop:request-local-helper`、`desktop:transcribe-local-media`、`local-asr:generate-course`、`desktop:open-external-url`，说明桌面 runtime bridge 已经齐全
- `app/infra/runtime_tools.py` 与 `desktop-client/electron/helper-runtime.mjs`: 已有 `yt-dlp / ffmpeg` 运行时查找和打包路径解析

### Established Patterns
- 桌面端能力通过 Electron main/preload/helper 注入共享前端，不另起第二套桌面 UI
- 上传与生成已经共用一个状态机式表面，链接导入被设计为进入相同课程生成闭环，而不是独立产品
- 既有产品方向强调低技术感知，用户不应被暴露给 helper、模型、ffmpeg、yt-dlp 等概念
- 当前 Phase 4 已经支持公开链接导入和失败 fallback，因此 Phase 07.1 更像“产品化与边界收口”，不是“第一次把功能做出来”

### Integration Points
- 任何 Memo 模式复刻都应优先落在 `UploadPanel.jsx` 的共享上传表面，而不是新增桌面专页
- 下载/媒体准备/课程生成的分层应围绕 `desktop_asr.py`、`runtime_tools.py`、`main.mjs` 现有职责边界收口
- 文案、平台承诺和桌面边界需要继续服从 Phase 7 的 Bottle 1.0 / 2.0 命名和 CTA 规范

<specifics>
## Specific Ideas

- 这次“参考 Memo”不是要把 Bottle 变成完整媒体下载器，而是复制其正式产品链路感：链接输入后，用户感知到的是一条完整稳定的桌面课程生成路径。
- 参考实现可继续人工对照本机安装的 Memo 包与日志，但不把它当作可直接复制源码的依赖来源；当前可确认的是能力边界和服务分层，而不是源码级复用。
- 对外举例时可使用少量常见平台示例，例如 `YouTube`、`B站`、`常见播客页面`、`公开视频直链`，帮助用户建立预期。
- 标题是本阶段唯一应稳定利用的导入元数据，产品重点是让课程标题更自然，而不是把来源信息做成新的主界面信息架构。

</specifics>

<deferred>
## Deferred Ideas

- `cookies / 浏览器 cookie / 账号登录 / 会员或受限内容导入` — 本阶段明确不做，后续如要做应单独立项
- `yt-dlp` 更新入口、下载源选择、高级媒体工具面板 — 不符合本阶段低技术感知原则
- 封面、频道、作者、来源平台等媒体卡片化展示 — 容易把 phase 扩展成媒体资产产品化，应单独评估

</deferred>
