# Phase 07.1: Desktop Workflow Spec

**Phase:** 07.1-memo  
**Status:** Draft for execution  
**Last updated:** 2026-03-29

## 用户主流程

Phase 07.1 固定的正式桌面链路是：

`粘贴公开链接 -> 本地下载/媒体准备 -> 课程生成 -> 历史/学习`

这条链路必须继续满足以下约束：

- `不新增桌面专页`
- `继续复用 UploadPanel`
- `成功后默认直达学习`
- 用户感知到的是一条完整桌面学习工作流，而不是多个独立 helper 步骤

## 阶段状态

用户可感知状态固定为以下几类：

- `等待导入`
  说明：用户还未提交链接，或已经回到待命状态准备下一次导入。
- `正在下载素材`
  说明：公开链接已经进入本地下载或媒体准备阶段，仍处于一个连续工作流中。
- `课程生成中`
  说明：素材准备完成后进入既有课程生成闭环，不出现第二套独立导入产品语义。
- `已取消链接下载`
  说明：用户主动中止当前导入任务，允许留在当前表面继续调整或重试。
- `链接导入失败，请稍后重试`
  说明：公开链接导入失败时的基础失败态，后续可根据受限或不支持分类给出更明确的辅助说明。

## 标题与元数据

本阶段链接导入后的元数据使用原则固定如下：

- `只稳定利用 title 作为课程默认标题来源`
- `标题优先于通用占位名`
- 当用户手动编辑标题时，用户输入优先于抓取标题
- `不把封面、作者、频道、来源平台升级成新的主界面资产`
- 来源平台、作者、频道、封面等信息如需保留，只能作为内部诊断或补充数据，不能变成新的用户主流程信息架构

## 成功与失败恢复

成功和失败恢复路径固定如下：

- 导入成功后进入既有 lesson/history/learning 闭环，不创建单独“链接导入历史”
- `失败后保留当前链接，便于重试`
- `失败分流优先推荐 SnapAny`
- `限制/登录态场景必须明确失败，而不是转向绕过能力建设`
- 当任务被取消时，用户仍应能继续调整标题、替换链接或切回本地文件流程

## 下游消费面

以下文件必须直接消费这份工作流规范：

- `frontend/src/features/upload/UploadPanel.jsx`
  负责用户可见导入表面、状态切换、失败恢复、标题联动、成功后跳转学习。
- `app/api/routers/desktop_asr.py`
  负责公开链接导入任务、错误分类、下载/媒体准备任务状态与标题元数据返回。
- `desktop-client/electron/main.mjs`
  负责 renderer 与本地 helper 的桥接、诊断内收和桌面运行时集成。
- `desktop-client/electron/preload.cjs`
  负责向 renderer 暴露必要桥接，但不扩展成普通用户可见媒体工具面板。
- `Docx/产品介绍.md`
  负责对外说明桌面链接导入能力，必须与本规范保持一致。
