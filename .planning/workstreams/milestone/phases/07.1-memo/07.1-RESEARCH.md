# Phase 07.1: Memo 模式复刻与桌面媒体工作流产品化 - Research

**Researched:** 2026-03-29  
**Domain:** Electron 桌面端公开链接导入、本地媒体准备、课程生成与学习闭环产品化  
**Confidence:** HIGH

## Summary

Phase 07.1 不是从零开始做“链接导入”，而是把已经存在的桌面 helper 能力收口成正式产品能力。

当前代码已经具备真实链路：

- `frontend/src/features/upload/UploadPanel.jsx` 已有 `链接导入` 入口、`导入并生成课程` 主按钮、下载进度、取消、失败提示、`SnapAny` fallback、标题联动、成功后直达学习
- `app/api/routers/desktop_asr.py` 已有 `/api/desktop-asr/url-import/tasks`、轮询、取消、文件回取、`yt-dlp` 元数据探测与下载错误分类
- `desktop-client/electron/main.mjs` 与 `desktop-client/electron/preload.cjs` 已有桌面 helper bridge、外链打开、helper 诊断与本地文件桥接
- `desktop-client/electron/helper-runtime.mjs`、`desktop-client/package.json`、`tests/contracts/test_desktop_installer_contract.py` 已覆盖 `yt-dlp / ffmpeg` 打包与运行时解析

因此本阶段的核心不是“再发明一个下载器”，而是把以下产品 contract 正式锁定并补齐：

1. Memo 模式复刻到底复制什么，不复制什么  
2. 对外支持承诺如何表述，避免滑向“支持所有 yt-dlp 站点”  
3. 用户只看到什么控制面，哪些诊断/工具能力必须继续藏在内部  
4. 链接导入后哪些元数据能进入主流程，哪些不能  
5. 自动化和人工回归如何证明“这已经是可发布的桌面工作流”，而不是分散 helper 行为

**Primary recommendation:** 将 Phase 07.1 拆成 3 个 plans：

1. `07.1-01` — 先固定桌面工作流规范和支持边界文档
2. `07.1-02` — 再把 UploadPanel / helper / runtime / 产品介绍统一到这个 contract
3. `07.1-03` — 最后用 unit / contract / integration / e2e + 手工 checklist 锁定发布边界

## User Constraints (from CONTEXT.md)

- 只复制 Memo 的公开链接工作流，不做 `cookies / 账号登录 / 会员内容 / 受限内容导入`
- 桌面端能力边界固定为：常见公开视频页链接 + 常见公开直链
- 默认用户表面只暴露：链接输入、课程标题、进度、取消、失败提示、后续生成动作
- 不向普通用户暴露 `yt-dlp` 更新、下载源选择、cookie 管理、登录态管理或高级媒体工具面板
- 对外承诺采用少量明确示例平台，而不是 `yt-dlp` 能力口径
- 对外文案应表达为 `支持常见公开视频链接`
- 示例用于设定预期，不构成全量平台兼容清单
- 稳定利用的元数据只有 `title`
- 封面、作者、频道、来源平台不升级成新的主界面资产
- 链接导入最终目标是形成正式桌面课程生成链路，而不是把 Bottle 变成媒体下载器

</user_constraints>

## Existing Architecture

### Frontend surface

`frontend/src/features/upload/UploadPanel.jsx`

已知能力：

- `desktopSourceMode === "link"`
- `导入并生成课程`
- `SnapAny` fallback
- `onNavigateToLesson?.(...)` 直达学习
- 链接失败重试、取消下载、后台继续
- 课程标题与链接元数据联动

关键结论：

- Phase 07.1 不需要新建桌面专页
- Memo 模式复刻必须继续落在现有上传面板，而不是再起第二套“媒体工具”信息架构

### Helper / backend path

`app/api/routers/desktop_asr.py`

已知能力：

- `_sanitize_source_url()` 会从带噪分享文案提取第一条有效 URL
- `_probe_ytdlp_metadata()` / `_download_media_with_ytdlp()` 已能处理公开视频页链接
- `URL_IMPORT_RESTRICTED` / `URL_IMPORT_UNSUPPORTED` / `URL_IMPORT_INVALID_URL` / `URL_IMPORT_CANCELLED` 已有基础错误分类
- 返回字段已经包含 `title`、`webpage_url`、`extractor_key`

关键结论：

- 后端链路已经足够支撑“公开链接优先”的产品化
- 需要收口的是公开边界、消息 contract、以及“只把标题用于主流程”的使用规范

### Desktop runtime

`desktop-client/electron/main.mjs`  
`desktop-client/electron/preload.cjs`

已知能力：

- renderer 通过现有 helper bridge 请求本地接口
- 诊断、外链打开、本地文件桥接都已存在
- 打包运行时已经带 `yt-dlp / ffmpeg`

关键结论：

- 本阶段不需要新增普通用户可见的高级控制桥接
- 诊断仍应留在 `desktop:get-helper-status` / `desktop:request-local-helper` 等内部面板能力

## Patterns to Reuse

### 共享上传表面

Phase 04 已经证明：

- 链接导入属于共享上传/生成面板
- 用户可接受一条链路完成下载、准备、生成、学习
- `SnapAny` 作为 secondary fallback 已是既有模式

### 桌面优先重媒体能力

项目现有产品边界已经固定：

- 重媒体处理留在本地
- 网页端不能假装支持 Bottle 1.0 或浏览器内链接导入
- 桌面端是完整体验，网页端是安全子集

### 标题是唯一稳定元数据资产

代码目前已经返回 `title`，这正好契合 Phase 07.1 决策。后续实现应避免扩成封面/作者/平台卡片系统。

## Common Pitfalls

### Pitfall 1: 把对外承诺写成工具能力

如果文案开始按 `yt-dlp 可解析` 表述，用户会自然推导出“几乎什么都支持”，这与当前 Phase 07.1 的公开视频边界直接冲突。

### Pitfall 2: 复刻 Memo 时把高级控制面一起搬进来

如果新增 cookie、登录态、下载源选择、更新入口等 UI，本阶段会从“产品化”滑回“工具化”。

### Pitfall 3: 让平台/作者/封面进入主流程

这会把阶段目标从“更顺滑地产生课程”扩成“媒体资产管理产品”，超出范围。

### Pitfall 4: 只改文档不锁测试

当前已有很多能力是通过源码/contract tests 暗含的；如果不补 Phase 07.1 级别的验证，后续阶段很容易把边界又改散。

### Pitfall 5: 把受限内容失败写成“再试试高级设置”

本阶段锁定的是明确失败 + 推荐 `SnapAny`，不是引导用户去调底层工具。

### Gaps to close

1. 公开链接支持示例与“不支持 cookies / 登录 / 会员内容”的产品文案 contract
2. 标题是唯一稳定元数据资产，而封面/作者/平台不进入主流程的约束
3. helper/runtime 仍无 cookie 管理、下载源选择等桥接的 contract
4. 产品介绍与桌面端真实行为的一致性
5. 手工发布前检查：公开视频成功、受限内容明确失败、诊断仍是内部入口

## Key Recommendation

不要把 Phase 07.1 计划成“新增一个更强的链接下载器”。

应该计划成：

- 先写清楚 Memo 模式复刻的正式产品 contract
- 再让 UploadPanel、desktop helper、runtime bridge、产品介绍统一遵守这个 contract
- 最后用自动化与手工 checklist 锁定“公开视频优先、低技术感知、标题最小元数据、受限内容明确失败”的发布边界
