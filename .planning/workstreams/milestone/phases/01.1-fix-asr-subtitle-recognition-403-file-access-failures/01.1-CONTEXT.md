# Phase 01.1: fix-asr-subtitle-recognition-403-file-access-failures - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Stabilize the existing Bottle 2.0 DashScope direct-upload lesson pipeline when subtitle recognition fails with cloud file-access `403` errors. This phase is about repairing the current cloud-file transcription path so the uploaded object remains usable for ASR task execution; it does not add a new generation mode, new fallback product surface, or a new media-processing route.

</domain>

<decisions>
## Implementation Decisions

### 403 recovery path
- **D-01:** When the DashScope direct-upload lesson path fails with `ASR_TASK_FAILED` and subtask code `FILE_403_FORBIDDEN`, the backend should transparently self-heal once by fetching a fresh signed URL from the canonical `dashscope_file_id` and retrying the ASR task one time.
- **D-02:** The retry should happen server-side and remain invisible to the user unless the second attempt also fails; manual re-upload is not the primary fix for this phase.
- **D-03:** Recovery should continue to treat `dashscope_file_id` as the canonical cloud object key. Client-provided `dashscope_file_url` remains compatibility fallback input, not the primary recovery key.

### Failure classification and observability
- **D-04:** `FILE_403_FORBIDDEN` should be treated as a first-class cloud file-access failure mode in the direct-upload path, not collapsed into a generic Bottle 2.0 outage during backend handling.
- **D-05:** The task/debug artifacts should record structured recovery details for this path, including the `dashscope_file_id`, first failure stage/code/message, whether a retry was attempted, and the retry/final outcome.

### the agent's Discretion
- Exact retry timing and whether it is immediate or uses a short backoff
- Exact debug-field names and where they are stored in task failure artifacts
- Exact user-facing copy after the self-heal retry is exhausted, as long as it reflects a cloud file-access failure rather than a generic outage

</decisions>

## Existing Code Insights

### Reusable Assets
- `app/services/lesson_service.py`: already owns the DashScope direct-upload lesson path, including signed URL resolution and ASR invocation
- `app/infra/asr_dashscope.py`: already exposes the exact `task_status`, `subtask_code`, and `subtask_message` data needed to detect `FILE_403_FORBIDDEN`
- `app/services/lesson_command_service.py`: already persists task artifacts and failure debug fields through `mark_task_failed(...)`
- `frontend/src/features/upload/UploadPanel.jsx`: already submits the canonical `dashscope_file_id` after direct upload succeeds

### Established Patterns
- The product already treats `dashscope_file_id` as the canonical cross-layer identifier for cloud-uploaded media
- The direct-upload path is split into three steps: request upload policy, upload file to DashScope OSS, then create the lesson task
- Lesson generation tasks already preserve structured task/debug data and expose task failure state back to the UI
- Phase 01 explicitly chose to fix the cloud-file path itself rather than silently diverting users into server-side media processing

### Integration Points
- Self-heal retry logic should land in the backend direct-upload execution path around `LessonService.generate_from_dashscope_file_id()` and/or `transcribe_signed_url()`
- Structured observability should connect to existing task artifact/failure storage in `lesson_command_service.py` and task manager outputs
- Frontend changes, if any, should stay limited to surfacing the more accurate failure classification after backend retry exhaustion

</code_context>

<specifics>
## Specific Ideas

- The user explicitly wants the system to solve the `403` problem rather than optimize fallback paths first.
- This phase should repair the canonical direct-upload ASR chain itself; fallback behavior from Phase 01 is not the primary implementation goal here.
- The preferred recovery is a transparent backend self-heal using a fresh signed URL derived from the canonical file ID.

</specifics>

<deferred>
## Deferred Ideas

- Broader fallback-product decisions remain governed by Phase 01 and later phases; they are not the focus of this urgent repair.
- Additional preflight validation or heavier object-health checks before retry were considered but deferred in favor of the simpler one-time transparent self-heal.

</deferred>
