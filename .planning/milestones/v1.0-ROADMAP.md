## Overview

v1.0 stabilizes the shared cloud generation path for Bottle 2.0, makes the desktop client the full-capability experience with Bottle 1.0 local generation, and fixes critical DashScope file-access failure recovery. The milestone achieves a clean runtime split: web users get a reliable Bottle 2.0 cloud path with transparent self-healing, and desktop users get a zero-friction local generation experience.

## Milestone Summary

**Decimal Phases:**

- Phase 01.1: Fix ASR subtitle recognition 403 file access failures (inserted after Phase 1 for urgent fix)

**Key Decisions:**

- Treat `dashscope_file_id` as the canonical cloud object key across request-url, task artifacts, and generation entrypoints. The direct-upload path already works end-to-end when the same object key is preserved; locking that contract in tests prevents the request-url response and task creation path from drifting apart. — Phase 01
- Retry direct-upload ASR exactly once only for `ASR_TASK_FAILED` plus `FILE_403_FORBIDDEN`, refreshing the signed URL from the canonical `dashscope_file_id`. — Phase 01.1
- Persist `dashscope_recovery` metadata on recovered lessons and map exhausted retries to `DASHSCOPE_FILE_ACCESS_FORBIDDEN` instead of a generic cloud outage. — Phase 01.1
- For browser direct-upload, return `oss://<file_id>` from the upload-policy response, prefer that resource URL for ASR task creation, and enable `X-DashScope-OssResourceResolve` instead of requiring `Files.get()` to expose an HTTPS signed download URL. — Post Phase 01.1
- Desktop client is the complete capability surface; web centers on Bottle 2.0 and browser-safe capabilities. — Project init
- Platform-managed billing/keys replace user-managed ASR secrets. — Project init
