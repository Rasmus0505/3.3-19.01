## 必须遵守

### 既有改动处理

检测到一批不是本次任务产生的现有改动时，忽略这些既有改动，继续只做本次任务

### 修改原则

- **存量优先**：优先在现有逻辑上扩展，而非新增文件。若必须新增，请简述理由。
- **影子清理**：修改文件时，顺手清理该文件内的死代码和拼写错误，但不要扩散到其他文件。
- **文件删除确认**：文件删除经过我确认后才能执行。
- **调试日志策略**：
  - 自动在关键逻辑处添加 `[DEBUG]` 标签的日志。
  - **自动销毁**：当你发现我开始下一个任务，或者我反馈功能已 OK 时，在下一次代码输出中**主动删除**之前添加的所有调试信息。
- **验证要求**：输出代码前，自检是否会报错。
- **报错信息**：严格保留完整的报错信息，不要为了兜底或者正常运行隐藏报错信息。

### 代码与部署硬约束

1. 不要破坏现有 API 契约
   - 路径：`/api/transcribe/file`
   - 核心字段不要随意改名

2. 不要硬编码 key
   - key 只允许环境变量

3. 不要把部署复杂化
   - 默认 Zeabur 一键可部署
   - 不新增用户难以理解的运维步骤
   - 主应用对外端口统一按 `8080` 说明和交付；给 Zeabur 的部署指引、环境变量建议、验收说明默认写 `PORT=8080`
   - 后续回答启动端口或部署端口时，不再把 `8000` 作为推荐值；若发现代码默认值与此不一致，应先说明“规则要求是 8080，代码现状需单独修正”

4. 任何重要改动都要有回归验证
   - 至少验证：`/health` + 上传转写成功

5. 数据库可视化默认约束
   - Metabase 只同步业务 schema：`app`
   - 不要把 Metabase 系统表和业务表混在同一个浏览入口

## 默认假设

### 项目部署方式

这个项目是用户将代码上传到github，然后Zeabur 读取github上的代码，自动部署到Zeabur上，不是用户自己买服务器、自己运维的方案。  
请默认：

- 由 Zeabur 负责运行容器
- 用户只会让 Zeabur ai配置环境变量和部署
- 不要要求用户做复杂服务器操作（Nginx、PM2、Linux 运维等）

### 用户画像

目标用户不是运维工程师，特征：

- 会用 GitHub、Zeabur
- 不懂服务器部署细节
- 希望“少配置、少命令、能跑通”

## 输出模板

### 给用户的回答应优先是

- 这一步做什么
- 为什么做
- 做完怎么验证

### 最后提交内容要额外加上下面三点

1. 本地更新内容（不需要带文件名）
2. 如果下次要完成同样的任务提示词怎样更科学，把这个科学的提示词总结出来方便用户备注，并在提示词后面单独为本次任务拟一个小标题，以后好回溯快速开始这次同样任务，这个提示词只负责描述这次具体任务，不需要可以提醒“请先读取 Docx/AI分工 下全部 md，避开其他 AI 正在修改的文件”这样的早就写在agent.md的广泛要求
3. 需要我在zeabur上采取什么操作（如果只是常规重新部署无需提示）

### 交付补充

- **交付提示**：要告诉用户接下来要去zeabur部署什么服务或者环境变量，以及有没有未完成任务
- 可以让用户复制你给他发的可以发给zeabur ai的提示词帮忙部署
