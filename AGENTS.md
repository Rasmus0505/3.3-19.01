## 必须遵守

### 既有改动处理

检测到一批不是本次任务产生的现有改动时，忽略这些既有改动，继续只做本次任务

### 修改原则

- **存量优先**：优先在现有逻辑上扩展，而非新增文件。若必须新增，请简述理由。
- **影子清理**：修改文件时，顺手清理该文件内的死代码和拼写错误，但不要扩散到其他文件。
- **文件删除确认**：文件删除经过我确认后才能执行。
- **调试日志策略**：
  - 自动在关键逻辑处添加 `[DEBUG]` 标签的日志。
  - **自动销毁**：当你发现我开始下一个任务，或者我反馈功能已 OK 时，在下一次代码输出中**主动删除**之前添加的所有调试信息。
- **验证要求**：输出代码前，自检是否会报错。
- **报错信息**：严格保留完整的报错信息，不要为了兜底或者正常运行隐藏报错信息。

### 代码与部署硬约束

1. 不要破坏现有 API 契约
   - 路径：`/api/transcribe/file`
   - 核心字段不要随意改名

2. 不要硬编码 key
   - key 只允许环境变量

3. 不要把部署复杂化
   - 默认 Zeabur 一键可部署
   - 不新增用户难以理解的运维步骤
   - 主应用对外端口统一按 `8080` 说明和交付；给 Zeabur 的部署指引、环境变量建议、验收说明默认写 `PORT=8080`
   - 后续回答启动端口或部署端口时，不再把 `8000` 作为推荐值；若发现代码默认值与此不一致，应先说明“规则要求是 8080，代码现状需单独修正”

4. 任何重要改动都要有回归验证
   - 至少验证：`/health` + 上传转写成功

5. 数据库可视化默认约束
   - Metabase 只同步业务 schema：`app`
   - 不要把 Metabase 系统表和业务表混在同一个浏览入口

## SSH / SFTP 远程操作规范

### 生效条件

- 只有在用户明确授权“允许直连 SSH / SFTP”后，才可以执行远程操作
- 默认仍优先使用 GitHub + Zeabur 的常规部署方式；SSH 只作为例外能力
- 未明确授权前，只能给方案、清单、命令模板，不要直接连接远端

### 操作边界

- 远程操作默认只允许触达用户明确指定的主机、目录、文件
- 默认先做只读检查，再做最小写入；不要从“已连上服务器”直接跳到修改系统
- 默认禁止触碰 `k3s`、systemd、iptables、用户管理、系统包升级、系统配置、服务编排和其他系统层内容
- 默认不执行破坏性命令，不批量删除，不覆盖未知文件，不擅自重启服务
- 任何超出“上传文件 / 校验目录 / 只读排查”的动作，都要先向用户说明再执行

### 连接前必须确认

- 目标主机 / IP 或域名
- SSH 端口
- 用户名
- 认证方式：私钥或密码
- 允许操作的目标目录
- 本次是否允许写入、重命名、覆盖、删除

### SSH 连接流程（模板）

1. 先拿到连接所需 5 项：`host`、`port`、`user`、认证方式、允许操作目录
2. CLI 连接模板：
   - `ssh -i <private_key> <user>@<host> -p <port>`
   - 如用密码：`ssh <user>@<host> -p <port>`
3. SFTP 连接模板：
   - `sftp -i <private_key> -P <port> <user>@<host>`
4. Windows 图形客户端可使用 `WinSCP` 或 `FileZilla`
   - Host：`<host>`
   - Port：`<port>`
   - User：`<user>`
   - Auth：私钥或密码
5. 连接成功后，先执行只读验证，再决定是否继续写入

### 登录后的只读验证模板

- `whoami`
- `hostname`
- `pwd`
- `df -h`
- `mount | grep /data`
- `ls -lah /data`

若任务与模型上传有关，优先确认这两个目标目录：

- `/data/asr-models/SenseVoiceSmall`
- `/data/asr-models/faster-distil-small.en`

### 远程写操作规范

- 写操作前先说明“准备写入哪个目录、写入什么文件、是否覆盖旧文件”
- 优先保留旧文件；如有冲突，优先新目录、临时目录或显式对比后再替换
- 上传后至少校验：目标路径、文件数、文件大小；必要时再做 hash 校验
- 若发现挂载异常、权限异常、路径不符、已有未知文件，立即停止并汇报

### 敏感信息规则

- 连接信息只作为当前会话输入使用，不写入仓库，不落入文档，不回显完整私钥
- 如需记录命令，使用模板或脱敏值，不把真实密码、私钥、长期令牌写进文件

### 服务器重置与数据边界

- 服务器重置不等于数据库一定丢失，必须区分三类对象：
  - PostgreSQL 服务及其数据卷
  - `web` 服务挂载的 `/data` 持久卷
  - 独立服务器的系统盘
- 任何 SSH 操作前，先确认数据库是否独立托管、`/data` 是否独立挂载、是否已有备份或恢复路径
- 如果数据库与系统盘在同一台机器且没有独立卷或备份，必须先明确提示“重置可能导致数据库不可恢复”
- 如果只重置承载 `web` 的服务器，而数据库和持久卷独立存在，通常优先影响的是运行环境，不一定影响数据库数据

### SSH 任务交付要求

- 说明本次触达了哪些主机 / 目录
- 说明执行了哪些只读检查
- 说明执行了哪些写操作
- 明确说明未触碰哪些系统层对象
- 如有异常，说明停止点和回滚 / 恢复路径

## Production Migration Rule

- 如果任务涉及生产数据库迁移或 Alembic 迁移，默认不要再依赖 Zeabur 容器启动自动迁移。
- 默认把 `AUTO_MIGRATE_ON_START` 视为应该在 Zeabur 中设置为字符串 `0`；除非用户明确要求恢复容器启动自动迁移，否则不要改回 `1`。
- 本地 agent 执行生产迁移时，固定在仓库根目录运行：
  - `python scripts/run_prod_migration.py`
- 只检查生产 revision 时，固定运行：
  - `python scripts/run_prod_migration.py --check-only`
- 生产迁移脚本优先读取 `PROD_DATABASE_URL`，没有时才回退到 `DATABASE_URL`。
- 如果 `PROD_DATABASE_URL` 和 `DATABASE_URL` 都不存在，必须直接报错停止，不要猜测连接串，不要私自改用 SQLite。
- 如果用户提供了公网 PostgreSQL 地址并要求本地 agent 执行迁移，优先把连接串写入当前机器的 `PROD_DATABASE_URL`，然后再运行固定命令。
- 执行完生产迁移后，至少回报这三项：
  - 当前 Alembic revision
  - 是否已经到 `head`
  - 是否还需要用户去 Zeabur 重启 `web`
