# 项目协作总规范

本文件是 `D:/3.3-19.01` 的 agent 总入口规范。目标不是替代 skill 文件，而是把本项目真正稳定的协作方式、任务池路由、交付约束写清楚，减少 agent 自行发挥。详细规范见 `Docx/规范/` 下各模块文件。

## 1. 固定背景与默认假设

- 项目根目录 `D:/3.3-19.01`，协作根目录 `D:/3.3-19.01/Docx`
- 协作目录：`Docx` 下同时包含 `before.md`、`after.md`、`tasks` 的子目录
- 任务池：`<协作目录>/tasks`
- 本项目由 Zeabur 从 GitHub 拉取代码并自动部署，由 Zeabur 负责运行容器
- 如当前环境已具备 SSH、PostgreSQL 等直接连接能力，可以使用，但必须以不破坏线上服务为前提

## 2. 全局协作规则

### 2.1 处理既有改动

- 发现一批不是本次任务产生的现有改动时，忽略这些既有改动，继续只做本次任务
- 不要回滚、覆盖或清理你没有产生的改动，除非用户明确要求
- 如果既有改动与当前任务直接冲突，先说明冲突点，再决定是否继续

### 2.2 修改策略

- 存量优先：优先在现有逻辑上扩展，而不是新增平行实现
- 若必须新增文件，需在最终交付中简述新增理由
- 修改文件时，可以顺手清理当前文件内直接相关的死代码、拼写错误、过时注释，但不要扩散到其他文件
- 删除文件必须先得到用户确认
- 不要吞掉错误、伪造成功或隐藏真实报错信息
- 输出前要自检：代码至少应保持语法、导入、引用关系可成立

### 2.3 调试与日志

- 只在确有定位问题需要时添加调试日志，不把临时调试当成默认长期实现
- 临时调试日志统一带 `[DEBUG]` 标签，便于后续搜索和清理
- 一旦问题定位完成，或开始进入下一个任务，应主动删除此前加入的调试信息

## 3. 任务池工作流总路由

本项目工作流程：`before` 和 `after` 是唯一标准入口。

### 3.1 什么时候用 `before`：计划任务

- 用户要把一个需求拆成任务池 YAML
- 用户要修复 `blocked` 任务定义
- 用户要重置任务池
- 用户认为某个已完成任务结果不对，需要把原任务转成修复轮次
- 用户在继续正式开发前，希望先规范规划任务池

`before` 只负责规划，不写实现代码，不替 `after` 干活。

### 3.2 什么时候用 `after`：从任务池执行任务

- 用户要求继续做任务池里的下一个任务
- 用户显式调用 `$after`
- 用户要求消费当前排队任务
- 用户要求继续修复已经进入 `fix` 的任务

`after` 只负责执行任务池现有任务，不新建任务，不重写任务目标，不绕过任务池直接把口头需求当实现单。

### 3.3 什么时候先不动任务池

- 用户在讨论 `before` 或 `after` skill 本身
- 用户在复盘某次规划是否合理
- 用户只是在问"现在应该做哪个任务"
- 用户在评价已交付结果，但真实诉求还没有说清

### 3.4 协作目录解析顺序

1. 如果用户明确指定了协作目录，直接使用
2. 否则扫描 `D:/3.3-19.01/Docx` 下所有子目录
3. 仅把同时包含 `before.md`、`after.md`、`tasks` 的子目录视为候选
4. 如果只有一个候选，直接使用它；如果有多个候选，使用最近修改的那个
5. 如果没有候选，停止并明确说明当前没有可用任务池

## 4. `before` 规划规范

详见 `Docx/规范/task-yaml-spec.md`（YAML 结构与字段规则）和 `Docx/规范/task-planning-rules.md`（规划流程与拆分规则）。

`before` 完成后，简短说明：使用了哪个协作目录、新建了哪些任务文件、修复了哪些 `blocked` 任务、把哪些已完成任务转成了 `fix`、哪些任务带有 `conflicts_with`、`after` 现在应从哪个任务开始、一句"本次关键决策：..."。

不要再额外输出一长段实现计划；任务池 YAML 是唯一事实来源。

## 5. `after` 执行规范

详见 `Docx/规范/task-execution-rules.md`。

`after` 完成后，简短说明：本轮完成了哪些任务、哪些任务被标记为 `blocked`、哪些任务因 `conflicts_with` 未解决而未启动、任务池是否仍有可执行工作、一段结果导向的本地更新摘要、`Zeabur操作：...`。

如果本轮至少完成了一个任务或修复任务，还必须额外提供：一个下次做同类任务可复用的中文提示词 + 约 8-12 个中文字符的短标题。

## 6. 最终交付统一要求

- 给出本地更新内容摘要，不需要按文件逐条罗列
- 如果需要给出"修改文件列表"，必须使用分叉树形式输出，按目录层级展开
- 告诉用户是否还有未完成任务、剩余风险或被阻塞项
- 如果下次要做同样任务，提供一个更科学、更聚焦当前具体任务的提示词，附约 8-12 个中文字符的短标题
- 单独给出一行 `Zeabur操作：...`

## 7. 数据库更新规则

以下情况必须执行 数据库迁移

- 改动涉及 Alembic migration、SQLAlchemy model、业务表清单或必需列
- 改动涉及删表、加表、改列，或任何需要把本地数据库结构同步到线上 PostgreSQL 的行为

执行原则：只要当前环境具备可用的生产数据库连接信息，就应先检查、再执行、再直连验证。只有在缺少生产数据库连接信息、用户明确禁止连接线上数据库、或迁移失败且无法自修时，才允许把数据库更新留给用户手动处理，但必须明确说明阻塞原因。

如果本次任务不需要数据库更新，最终交付里必须明确写出：`本次无需执行数据库迁移`。

## 8. 任务池是唯一事实来源

一旦进入 `before` / `after` 工作流，任务池 YAML 是唯一事实来源。`before` 负责定义任务，`after` 负责消费任务。不要在聊天里临时发明与任务池冲突的新边界、新状态、新字段或新验收。若真实目标变了，应先修任务池，再继续执行。
