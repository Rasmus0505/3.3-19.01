# 项目协作总规范

本文件是 `D:/3.3-19.01` 的 agent 总入口规范。目标不是替代 skill 文件，而是把本项目真正稳定的协作方式、任务池路由、交付约束写清楚，减少 agent 自行发挥。

## 1. 固定背景与默认假设

- 项目根目录 `D:/3.3-19.01`，协作根目录 `D:/3.3-19.01/Docx`
- 协作目录：`Docx` 下同时包含 `before.md`、`after.md`、`tasks` 的子目录
- 任务池：`<协作目录>/tasks`
- 本项目由 Zeabur 从 GitHub 拉取代码并自动部署，由 Zeabur 负责运行容器
- 如当前环境已具备 SSH、PostgreSQL 等直接连接能力，可以使用，但必须以不破坏线上服务为前提

## 2. 全局协作规则

### 2.1 处理既有改动

- 发现一批不是本次任务产生的现有改动时，忽略这些既有改动，继续只做本次任务
- 不要回滚、覆盖或清理你没有产生的改动，除非用户明确要求
- 如果既有改动与当前任务直接冲突，先说明冲突点，再决定是否继续

### 2.2 修改策略

- 存量优先：优先在现有逻辑上扩展，而不是新增平行实现
- 若必须新增文件，需在最终交付中简述新增理由
- 修改文件时，可以顺手清理当前文件内直接相关的死代码、拼写错误、过时注释，但不要扩散到其他文件
- 删除文件必须先得到用户确认
- 不要吞掉错误、伪造成功或隐藏真实报错信息
- 输出前要自检：代码至少应保持语法、导入、引用关系可成立

### 2.3 调试与日志

- 只在确有定位问题需要时添加调试日志，不把临时调试当成默认长期实现
- 临时调试日志统一带 `[DEBUG]` 标签，便于后续搜索和清理
- 一旦问题定位完成，或开始进入下一个任务，应主动删除此前加入的调试信息

## 3. 任务池工作流总路由

本项目工作流程：`before`、`after` 和 `inspector` 是标准入口。

## 4. `before` 规划规范

详见 `Docx/协作/before.md`。

**触发条件**：用户要把一个需求拆成任务池 YAML / 用户要修复 blocked 任务 / 用户要重置任务池 / 用户认为已完成任务结果不对需要转修复轮次 / 用户希望先规范规划任务池。

`before` 只负责规划，不写实现代码，不替 `after` 干活。

**输出要求**：简短说明使用了哪个协作目录、做了几轮问答核心澄清了什么、新建了哪些任务文件（列出文件名）、哪些任务带有 `conflicts_with`（如果存在）、执行者现在从哪个任务开始。

任务池 YAML 是唯一事实来源。

## 5. `after` 执行规范

详见 `Docx/协作/after.md`。

**触发条件**：用户要求继续做任务池里的下一个任务 / 用户显式调用 `$after` / 用户要求消费当前排队任务 / 用户要求继续修复已经进入 `fix` 的任务。

`after` 只负责执行任务池现有任务，不新建任务，不重写任务目标，不绕过任务池直接把口头需求当实现单。

**输出要求**：简短说明本轮完成了哪些任务、哪些任务被标记为 `blocked`、哪些任务因 `conflicts_with` 未解决而未启动、任务池是否仍有可执行工作、一段结果导向的本地更新摘要、`Zeabur操作：...`。

如果本轮至少完成了一个任务或修复任务，还必须额外提供：一个下次做同类任务可复用的中文提示词 + 约 8-12 个中文字符的短标题。

## 5.5 `inspector` 审查规范

详见 `Docx/协作/inspector.md`。

**触发条件**：用户要求审查任务记录表 / 用户要求对已完成任务进行质量检查。

`inspector` 负责读取任务行为记录、判断任务是否正确完成，并将需求原始目标追加到产品修改历史。

## 6. 最终交付统一要求

- 给出本地更新内容摘要，不需要按文件逐条罗列
- 如果需要给出"修改文件列表"，必须使用分叉树形式输出，按目录层级展开
- 告诉用户是否还有未完成任务、剩余风险或被阻塞项
- 如果下次要做同样任务，提供一个更科学、更聚焦当前具体任务的提示词，附约 8-12 个中文字符的短标题
- 单独给出一行 `Zeabur操作：...`

## 7. 数据库更新规则

以下情况必须执行 数据库迁移

- 改动涉及 Alembic migration、SQLAlchemy model、业务表清单或必需列
- 改动涉及删表、加表、改列，或任何需要把本地数据库结构同步到线上 PostgreSQL 的行为

执行原则：只要当前环境具备可用的生产数据库连接信息，就应先检查、再执行、再直连验证。只有在缺少生产数据库连接信息、用户明确禁止连接线上数据库、或迁移失败且无法自修时，才允许把数据库更新留给用户手动处理，但必须明确说明阻塞原因。

如果本次任务不需要数据库更新，最终交付里必须明确写出：`本次无需执行数据库迁移`。

## 8. 任务池是唯一事实来源

一旦进入 `before` / `after` 工作流，任务池 YAML 是唯一事实来源。`before` 负责定义任务，`after` 负责消费任务。不要在聊天里临时发明与任务池冲突的新边界、新状态、新字段或新验收。若真实目标变了，应先修任务池，再继续执行。

## 8.5 GSD 规划文档默认中文

当本项目使用 GSD 生成或更新 `.planning/` 下的规划文档时，默认规则如下：

- 所有**面向人的规划内容**默认使用简体中文，包括标题、段落、说明、决策、成功标准、研究结论、计划说明、总结与验证说明
- 不要为了中文化去修改 GSD 依赖解析的结构字段
- 必须保持这些机器兼容结构原样：
  - YAML/frontmatter 字段名，例如 `phase`、`plan`、`requirements`、`wave`、`depends_on`、`files_modified`
  - XML/块标签，例如 `<context>`、`<decisions>`、`<verification>`、`<tasks>`、`<objective>`
  - REQ-ID、Phase/Plan 编号、命令、路径、接口路径、模型名、环境变量名
- 如果某个字段同时承担“给人看”和“给机器解析”的作用，优先保证 GSD 兼容，不要强行翻译字段名
- 对已有 `.planning/` 文档做增量更新时，延续已有中文风格，不要重新切回英文

目标：让本项目的 GSD 输出默认中文，同时不影响 GSD 工作流质量和兼容性。

## 9. 产品修改历史记录

### 9.1 记录目的

`Docx/协作/产品修改历史.md` 记录的是**用户的原始目标和第一性原理**，不是具体实现细节。

记录内容示例：
- ❌ 不要：`新增桌面端打包功能（Electron）`
- ✅ 要：`用户希望学生能离线使用 AI 辅导功能`

### 9.2 触发时机

当用户提出新需求时，AI 应先做需求澄清对话，精简出用户的原始目标和动机（第一性原理）。

工作流（before/after）结束后，任务记录表审查完毕后，由审查 AI（inspector）将本次需求的精简描述（核心目标 + 第一性原理）追加到 `Docx/协作/产品修改历史.md`。

### 9.3 格式

```markdown
## YYYY

### YYYY-MM-DD

- **方向**：一句话描述核心方向
- **原始目标**：用户想解决什么问题
- **用户动机**：用户的深层动机或背景（可选）
```

### 9.4 交互要求

收到新需求时，AI 应主动追问：
- 用户想解决的根本问题是什么？
- 这个需求的背景是什么？
- 有没有更简单的方式达到同样的目标？

不要急着接受用户的第一个解法，而是先理解用户的真实目标。
