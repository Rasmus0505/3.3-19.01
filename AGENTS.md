# 项目协作总规范

本文件是 `D:/3.3-19.01` 的 agent 总入口规范。目标不是替代 skill 文件，而是把本项目真正稳定的协作方式、任务池路由、交付约束、Zeabur 与数据库默认假设写清楚，减少 agent 自行发挥。

## 1. 固定背景与默认假设

- 项目根目录固定为 `D:/3.3-19.01`
- 协作根目录固定为 `D:/3.3-19.01/Docx`
- 协作目录是 `Docx` 下同时包含 `before.md`、`after.md`、`tasks` 的子目录
- 任务池固定为 `<协作目录>/tasks`
- 本项目默认由 Zeabur 从 GitHub 拉取代码并自动部署
- 默认由 Zeabur 负责运行容器，不假设用户自行购买服务器并手工运维
- 用户通常只会通过 Zeabur 配置环境变量、触发部署或查看服务状态，不做复杂服务器操作
- 如当前环境已具备 SSH、PostgreSQL 等直接连接能力，可以使用，但必须以不破坏线上服务为前提

## 2. 全局协作规则

### 2.1 处理既有改动

- 发现一批不是本次任务产生的现有改动时，忽略这些既有改动，继续只做本次任务
- 不要回滚、覆盖或清理你没有产生的改动，除非用户明确要求
- 如果既有改动与当前任务直接冲突，先说明冲突点，再决定是否继续

### 2.2 修改策略

- 存量优先：优先在现有逻辑上扩展，而不是新增平行实现
- 若必须新增文件，需在最终交付中简述新增理由
- 修改文件时，可以顺手清理当前文件内直接相关的死代码、拼写错误、过时注释，但不要扩散到其他文件
- 删除文件必须先得到用户确认
- 不要为了“看起来正常”而吞掉错误、伪造成功或隐藏真实报错信息
- 输出前要自检：代码至少应保持语法、导入、引用关系可成立

### 2.3 调试与日志

- 只在确有定位问题需要时添加调试日志，不把临时调试当成默认长期实现
- 临时调试日志统一带 `[DEBUG]` 标签，便于后续搜索和清理
- 一旦问题定位完成，或开始进入下一个任务，应主动删除此前加入的调试信息
- 不要把调试日志扩散到与当前任务无关的文件

### 2.4 讨论与执行分离

- 如果用户在讨论 skill 自身、回顾上一次规划/执行质量、质疑任务拆分、或只是询问是否应该运行某任务，先回答讨论，不要立刻改任务池或开始执行
- 只有当用户最新一条消息明确要求创建任务、修复任务、重置任务池、执行排队任务或继续处理任务池时，才进入任务池变更或执行流程

## 3. 任务池工作流总路由

本项目涉及任务池时，`before` 和 `after` 是唯一标准入口。

### 3.1 什么时候用 `before`

以下情况走 `before`：

- 用户要把一个需求拆成任务池 YAML
- 用户要修复 `blocked` 任务定义
- 用户要重置任务池
- 用户认为某个已完成任务结果不对，需要把原任务转成修复轮次
- 用户在继续正式开发前，希望先规范规划任务池

`before` 只负责规划，不写实现代码，不替 `after` 干活。

### 3.2 什么时候用 `after`

以下情况走 `after`：

- 用户要求继续做任务池里的下一个任务
- 用户显式调用 `$after`
- 用户要求消费当前排队任务
- 用户要求继续修复已经进入 `fix` 的任务

`after` 只负责执行任务池现有任务，不新建任务，不重写任务目标，不绕过任务池直接把口头需求当实现单。

### 3.3 什么时候先不动任务池

以下情况先只讨论，不变更任务池也不做实现：

- 用户在讨论 `before` 或 `after` skill 本身
- 用户在复盘某次规划是否合理
- 用户只是在问“现在应该做哪个任务”
- 用户在评价已交付结果，但真实诉求还没有说清

### 3.4 协作目录解析顺序

当需要进入任务池流程时，按下面顺序解析协作目录：

1. 如果用户明确指定了协作目录，直接使用
2. 否则扫描 `D:/3.3-19.01/Docx` 下所有子目录
3. 仅把同时包含 `before.md`、`after.md`、`tasks` 的子目录视为候选
4. 如果只有一个候选，直接使用它
5. 如果有多个候选，使用最近修改的那个，并在输出里说明
6. 如果没有候选，停止并明确说明当前没有可用任务池

## 4. `before` 规划规范

### 4.1 核心职责

`before` 负责：

- 读取用户当前需求
- 读取当前任务池
- 检查未完成任务是否已覆盖或冲突
- 检查本次需求相关的最小仓库上下文
- 将需求拆成可由单个 `after` 独立完成的任务 YAML
- 修复可修复的 `blocked` 任务定义
- 在用户明确否定已交付结果时，把原 `done` 任务转换为 `fix`

### 4.2 禁止事项

- 不写实现代码
- 不修改任务池之外的项目实现文件
- 不代替 `after` 执行任务
- 不跳过任务池只输出聊天计划
- 不创建“只验证、不实现”的任务
- 不在高优先级模式禁止写入时强行写任务文件

### 4.3 写入受限时的行为

如果当前模式或上级规则禁止写入：

- 仍然可以读任务池和仓库上下文
- 只输出 dry-run 结果
- 明确说明本次没有写文件，原因是当前模式禁止变更

### 4.4 任务文件命名与状态

优先使用带中文短标题的文件名，兼容读取旧文件名。

- `TASK-001-中文短标题.yaml` 表示 `status: todo`
- `I-TASK-001-中文短标题.yaml` 表示 `status: in_progress`
- `B-TASK-001-中文短标题.yaml` 表示 `status: blocked`
- `F-TASK-001-中文短标题.yaml` 表示 `status: done`
- `fix-TASK-001-中文短标题.yaml` 表示 `status: fix`
- `fix-I-TASK-001-中文短标题.yaml` 表示 `status: fixing`

规则：

- 文件名里的中文后缀只用于可读性，`task_id` 必须保持 `TASK-001` 这类规范形式
- 读取现有任务时，需同时兼容带后缀与不带后缀的旧命名
- 新建可执行任务必须以 `TASK-...` 命名
- 计算下一个编号时，先去掉一个前缀：`fix-I-`、`I-`、`B-`、`F-`、`fix-`
- 再忽略文件名里 `TASK-编号` 之后的中文后缀
- 修复 `blocked` 任务使其重新可执行时，要改回 `status: todo`，清空 `blocked_reason`，并把文件名恢复为 `TASK-...`
- 把 `done` 任务转成修复任务时，要改成 `status: fix`，并把文件名改为 `fix-TASK-...`

### 4.5 任务 YAML 固定结构

```yaml
schema_version: 1
task_id: TASK-001
title: 中文任务标题
status: todo
priority: high
project_root: D:/3.3-19.01
base_ref: main
goal: 中文具体目标
scope:
  - 中文影响范围
allowed_paths:
  - app/api/example.py
acceptance:
  - 中文可观察验收标准 1
  - 中文可观察验收标准 2
check_commands:
  - pytest tests/example_test.py
depends_on: []
dependency_reason: ""
blocked_by: []
conflicts_with: []
notes_for_after: 给 after 的中文执行说明
result_summary: ""
changed_files: []
tests_run: []
blocked_reason: ""
completed_at: ""
fix_request: ""
fix_reason: ""
fix_round: 0
fix_result_summary: ""
fix_updated_at: ""
```

### 4.6 字段规则

- `schema_version` 必须始终为 `1`
- `task_id`、`depends_on`、`blocked_by`、`conflicts_with` 只能使用规范任务号，如 `TASK-001`
- 人工编写的规划内容统一写中文
- 新任务起始状态必须是 `todo`
- 只有把已完成任务转为修复轮次时，才能把状态写成 `fix`
- `priority` 只能是 `high`、`medium`、`low`
- `project_root` 必须始终是 `D:/3.3-19.01`
- `goal` 必须描述一个明确的实现结果
- `scope` 必须描述受影响的功能面或子系统
- `allowed_paths` 只允许使用仓库相对路径，且边界要足够明确，避免 `after` 猜范围
- 除非确实需要目录级边界，否则优先列具体文件
- `acceptance` 必须可观察、可验证，至少包含一个行为层验收条件
- 对既有行为的修改，至少补一个回归或安全验收条件
- `check_commands` 必须是真实可运行、且验证当前任务自身结果的命令
- 不要写占位命令，也不要把依赖于另一新任务输出的验证拆出去
- `depends_on` 默认为空；只有真实前置条件无法通过重画任务边界消除时，才允许非空
- `dependency_reason` 仅在确实存在依赖时填写，用中文解释为什么不能合并
- `conflicts_with` 只记录未完成任务；若无冲突，保持 `[]`
- brand new 任务的 `result_summary`、`changed_files`、`tests_run`、`blocked_reason`、`completed_at`、`fix_request`、`fix_reason`、`fix_result_summary`、`fix_updated_at` 必须为空
- brand new 任务的 `fix_round` 必须为 `0`

### 4.7 规划前的最小检查

在写或修任务前，至少检查这些内容：

- 与需求最相关的入口文件
- 附近的配置、schema、共享类型定义
- 该区域现有测试
- 如果改动涉及接口或契约，至少看一侧调用方或实现方

不要只凭用户一句话拍任务边界。

### 4.8 任务拆分规则

- 一个任务只服务一个清晰目标
- 一个任务必须能被单个 `after` 从实现、验证到回写独立做完
- 默认优先支持并行领取
- 若两个候选任务会修改同一高冲突面，默认合并，不要硬拆依赖

高冲突面包括但不限于：

- 路由入口
- 应用启动与全局配置
- 共享 API schema 或契约
- 数据库迁移
- 鉴权、会话、全局状态
- 跨模块共享类型
- `package.json`、lockfile
- 打包目标、安装器、更新机制、运行时捆绑策略
- 首次启动或安装流程

### 4.9 冲突与依赖检查

在写新任务或修任务前，必须扫描所有未完成任务：

- `todo`
- `in_progress`
- `blocked`
- `fix`
- `fixing`

检查依据：

- `allowed_paths`
- `scope`
- `goal`
- 是否共享高冲突面

规则：

- 如果当前需求已经被某个未完成任务覆盖，优先修或扩那个任务，不要重复新建
- 若必须创建独立新任务，但与未完成任务有冲突，必须把对应任务号写入 `conflicts_with`
- 不要因为另一个任务优先级低或当前是 `blocked` 就忽略冲突
- 依赖链默认不要超过一跳，除非用户明确要求分阶段推进

### 4.10 `blocked` 任务修复

发现 `blocked` 任务时，处理顺序如下：

1. 先读 `blocked_reason`
2. 把阻塞归类为：
   - 范围缺口
   - 路径权限缺口
   - 缺少前置任务
   - 目标歧义
   - 仓库漂移或状态不一致
3. 如果问题来自任务定义本身，原地修复原任务
4. 只有存在真实前置条件时，才新增一个支持任务
5. 如果修完已可执行，改回 `todo` 并清空 `blocked_reason`
6. 不要把“补全缺失定义”的责任留给 `after`

### 4.11 `done` 转 `fix`

当用户明确表示某个已完成任务结果不对、不完整或方向错了时：

1. 先定位原始 `done` 任务
2. 确认该任务当前状态是 `done`，文件名形如 `F-TASK-...`
3. 先判断属于哪类问题：
   - 同一交付目标下缺少实现细节
   - 同一主交付面下问题定义错了
   - 真正新增的相邻能力
4. 如果用户否定的是原任务的核心交付结果、最终形态或主要验收结果，优先把原任务转成 `fix`
5. 如果只是新增相邻能力，而原交付本身成立，则新建任务，不要改原任务
6. 转 `fix` 时保留原 `result_summary`、`changed_files`、`tests_run`、`completed_at`
7. `fix_round` 加 1
8. 用中文填写 `fix_request` 与 `fix_reason`
9. 更新 `fix_updated_at`
10. 重新计算 `conflicts_with`，只看当前未完成任务
11. 必要时重写 `title`、`goal`、`scope`、`acceptance`、`notes_for_after`，但不能顺势扩成无关新需求

### 4.12 `before` 输出要求

`before` 完成后，简短中文说明以下信息：

- 使用了哪个协作目录
- 新建了哪些任务文件
- 修复了哪些 `blocked` 任务
- 把哪些已完成任务转成了 `fix`
- 哪些任务带有 `conflicts_with`，对应冲突哪些任务号
- 是否仍存在依赖
- `after` 现在应从哪个任务开始
- 一句 `本次关键决策：...`
- 说明本次属于新建任务、修复 `blocked`、转换 `fix`，还是仅做 dry-run

不要再额外输出一长段实现计划；任务池 YAML 才是唯一事实来源。

## 5. `after` 执行规范

### 5.1 核心职责

`after` 负责：

- 扫描任务池
- 选择下一个可执行的未完成任务或修复任务
- 通过 subagent 实施代码、验证结果
- 把执行结果回写到原任务 YAML

### 5.2 禁止事项

- 不创建新任务
- 不改变任务目标
- 不绕过任务池，直接把口头需求当实现单
- 不擅自把 `blocked` 改回 `todo`
- 不擅自把 `done` 改成 `fix`
- 不在高优先级规则禁止写入时硬做实现

### 5.3 写入受限时的行为

如果当前模式禁止写入或执行：

- 仍可以读任务池并判断理论上的下一个任务
- 只返回 dry-run 结果
- 必须明确说明由于当前模式禁止变更，所以没有启动或恢复任务

### 5.4 状态机

`after` 只能使用下面六种状态：

- `todo`
- `in_progress`
- `blocked`
- `done`
- `fix`
- `fixing`

状态流转固定为：

1. 选中的 `todo` 先改成 `in_progress`，再把文件重命名为 `I-TASK-...`
2. 选中的 `fix` 先改成 `fixing`，更新 `fix_updated_at`，再把文件重命名为 `fix-I-TASK-...`
3. 已是 `in_progress` 或 `fixing` 的任务可以继续执行，不改目标
4. 成功完成后改成 `done`，保存后把文件重命名为 `F-TASK-...`
5. 无法安全继续时改成 `blocked`，保存后把文件重命名为 `B-TASK-...`

### 5.5 任务选择顺序

扫描任务池时：

1. 读取所有前缀形式：`TASK-...`、`I-TASK-...`、`B-TASK-...`、`F-TASK-...`、`fix-TASK-...`、`fix-I-TASK-...`
2. 跳过 `done`
3. 跳过 `blocked`
4. 必须满足 `depends_on` 全部已经 `done`
5. 必须满足 `conflicts_with` 中每个任务都已经 `done`
6. 如果 `conflicts_with` 未解决，不要更改当前任务状态和文件名，只向用户说明暂时不能安全启动
7. 若有多个可执行任务，优先顺序为：
   - `fixing`
   - `in_progress`
   - `fix`
   - `todo`
   - `priority: high`
   - `priority: medium`
   - `priority: low`
   - 去掉一个状态前缀后的规范文件名顺序

### 5.6 执行边界

- 只能修改 `allowed_paths` 内的文件
- 只实现当前任务的 `goal` 和 `acceptance`
- 如果当前任务处于 `fix` 或 `fixing`，还必须落实 `fix_request`
- 必须运行或确认 `check_commands`
- 除了当前任务 YAML 的状态回写与重命名，不要改其他任务文件
- 如果需要修改的耦合文件落在 `allowed_paths` 外，不能擅自越界，应直接标记 `blocked`

对于这些耦合面尤其要严格边界：

- 打包目标
- 安装器行为
- 更新机制
- bundled runtime 策略
- 首次启动 / 安装流程

### 5.7 subagent 规则

开始或恢复任务后，主线程必须先创建并使用至少一个 subagent，然后再进入实质工作。

规则：

- 主线程负责任务选择、改状态、改文件名、最终验收与回写
- subagent 负责调查、实现、修复、验证
- 每个 subagent 固定使用模型 `gpt-5.4`
- 只读检查可用 `low` 或 `medium`
- 会改代码、配置、迁移、脚本的工作默认 `high`
- 多文件高风险改动、修复轮次、打包/安装器/运行时策略改动默认 `xhigh`
- 只读调查优先 `explorer`
- 需要修改时优先 `worker`
- modifying subagent 必须被明确告知：它不是唯一在仓库里工作的 agent，不得回滚别人改动，不得超出当前任务边界

### 5.8 成功回写

任务成功后，必须在原任务文件中回写至少这些字段：

```yaml
status: done
result_summary: 中文完成摘要
changed_files:
  - 实际改动文件 1
  - 实际改动文件 2
tests_run:
  - 实际执行命令 1
  - 实际执行命令 2
completed_at: 2026-03-23T15:00:00+08:00
```

如果是修复任务，还要额外更新：

```yaml
fix_result_summary: 中文修复摘要
fix_updated_at: 2026-03-23T14:50:00+08:00
```

### 5.9 阻塞回写

若出现以下情况，必须停止并标记 `blocked`：

- 任务定义过于模糊，无法安全执行
- 关键必需文件不在 `allowed_paths`
- 缺少前置条件
- 仓库当前状态与任务定义冲突
- 缺少必需测试或运行前提，且无法合理准备

阻塞时写入：

```yaml
status: blocked
blocked_reason: >
  中文说明卡在哪一步、缺什么边界或前置条件，以及 before 需要补什么。
```

注意：

- 不要因为 `conflicts_with` 未解决就把任务标记为 `blocked`
- 这种情况要保持任务原状，只向用户报告“当前有冲突任务未完成，暂时不能安全执行”

### 5.10 工作循环与停止条件

`$after` 被调用来持续清任务池时，应循环执行，直到出现以下任一条件：

- 没有可执行的未完成任务
- 所有剩余未完成任务都是 `blocked`
- 所有剩余任务都在等待未完成依赖或未完成冲突任务

### 5.11 `after` 输出要求

`after` 完成一轮后，简短中文说明：

- 本轮完成了哪些任务
- 本轮完成了哪些修复任务
- 哪些任务被标记为 `blocked`
- 哪些任务因 `conflicts_with` 指向未完成任务而未启动或未恢复
- 任务池是否仍有可执行工作
- 一段结果导向的本地更新摘要
- 一行文件列表；若没有改动，写 `本次修改文件：无`
- 一行独立的 `Zeabur操作：...`
- 说明本次是正常执行、因 `blocked` 停止、因 `conflicts_with` 暂停，还是仅做 dry-run

如果本轮至少完成了一个任务或修复任务，还必须额外提供：

- 一个下次做同类任务可复用的中文提示词
- 紧跟提示词后给出一个约 8 到 12 个中文字符的短标题，便于以后回溯

## 6. 最终交付统一要求

所有 agent 的最终回复默认都要体现以下信息；若 `before` 或 `after` 有更严格输出格式，以对应工作流为准。

### 6.1 通用交付要求

- 给出本地更新内容摘要，不需要按文件逐条罗列
- 告诉用户是否还有未完成任务、剩余风险或被阻塞项
- 如果下次要做同样任务，提供一个更科学、更聚焦当前具体任务的提示词
- 在该提示词后单独给出一个本次任务短标题，便于以后快速回溯
- 这个提示词只描述本次具体任务目标、保留项、删除项、验证要求
- 不要把“先读 Docx 全部 md”“避开其他 AI 正在编辑的文件”这类已经写在总规范里的常驻要求塞进提示词
- 单独给出一行 `Zeabur操作：...`

### 6.2 Zeabur 交付说明

- 如果只需要常规 GitHub 触发重新部署或无需平台动作，写 `Zeabur操作：无`
- 如果需要 Zeabur 动作，明确写出是哪个服务、哪个环境变量、还是哪个数据库动作
- 不要笼统写“去 Zeabur 部署一下”

## 7. 数据库更新规则

以下情况必须执行 skill：`$postgresql-update`

- 改动涉及 Alembic migration
- 改动涉及 SQLAlchemy model
- 改动涉及业务表清单或必需列
- 改动涉及删表、加表、改列
- 改动涉及任何需要把本地数据库结构同步到线上 PostgreSQL 的行为

执行原则：

- 不要把“请你自己去 Zeabur 更新数据库”当默认收尾
- 只要当前环境具备可用的生产数据库连接信息，就应按 `$postgresql-update` 的流程先检查、再执行、再直连验证
- 只有在缺少生产数据库连接信息、用户明确禁止连接线上数据库、或迁移失败且无法自修时，才允许把数据库更新留给用户手动处理
- 若留给用户手动处理，必须明确说明阻塞原因
- 如果本次任务不需要数据库更新，最终交付里必须明确写出：`本次无需执行数据库迁移`

## 8. 任务池是唯一事实来源

一旦进入 `before` / `after` 工作流：

- 任务池 YAML 是唯一事实来源
- `before` 负责定义任务
- `after` 负责消费任务
- 不要在聊天里临时发明与任务池冲突的新边界、新状态、新字段或新验收
- 若真实目标变了，应先修任务池，再继续执行
