  - “开始生成课程”或同义动作是否明确
  - 进度条、阶段文案、等待反馈是否可信
  - 出错时是否知道原因、能否恢复
  - 上传中刷新页面后，任务是否丢失、状态是否恢复
  - 切到别的页面再回来时，任务提示、悬浮卡片、恢复横幅是否一致

  - 页面是否容易打断、卡住、失焦
  - 进度保存是否自然
  - 刷新或退出再进入后，是否能回到合理位置

### 高风险观察点
请特别留意并主动挑战这些地方：
- 上传任务状态是否前后不一致
- 生成完成后是否没有及时出现在历史记录
- 刷新页面后任务、登录、课程状态是否丢失
- 学习页媒体不可用时是否有合理降级
- 快捷键、输入焦点、自动播放、全屏是否互相打架
- 历史记录删除、重命名、多选是否有脏状态
- 钱包、价格、点数、余额文案是否混淆
- 管理台权限、导航、空态、报错是否对管理员友好
- 窄屏下按钮遮挡、滚动穿透、弹窗溢出、底部操作区不可见
- 成功提示过弱、失败提示过泛、等待过程缺少安全感

### 发现问题时的工作方式
每发现一个问题，不要只说“这里有 bug”。你必须补齐：
- 页面路径和页面名称
- 前置条件
- 最短复现步骤
- 期望结果
- 实际结果
- 严重级别：`P0` / `P1` / `P2` / `P3`
- 影响对象：新用户 / 老用户 / 管理员 / 付费用户 / 所有人
- 证据：截图、控制台报错、网络报错、录屏要点任选其一
- 问题归类：功能缺陷 / 易用性问题 / 信息架构问题 / 文案问题 / 性能问题 / 一致性问题 / 风险隐患
- 如果你能提出优化建议，必须写成可执行建议，不要只写“建议优化体验”

### 输出格式
最终输出必须只按下面结构给出，不要写成长篇散文。

1. 执行概览
- 本次使用了哪些身份、哪些视口、哪些关键路径
- 哪些路径验证完成，哪些因为条件不足未验证

2. 高优先级问题
- 每条问题一段，带严重级别、页面、复现步骤、期望、实际、影响、证据

3. 中低优先级问题
- 同上

4. 可优化点
- 每条写清楚“当前阻力是什么”“为什么影响转化、理解或完成率”“建议怎么改”

5. 暂未复现但值得警惕的风险
- 只列你根据实际观察推断出的高风险点，不要瞎猜

6. 结论
- 用 3 到 8 句话总结：这个网站目前最影响用户成功的地方是什么，最值得先修的 3 件事是什么

### 执行纪律
- 不要为了完成任务而跳过等待、跳过加载、跳过登录状态变化。
- 不要把接口 200 当作用户成功。
- 不要把“你理解了”当作“用户能理解”。
- 不要因为知道源码或产品意图，就替页面自己脑补。
- 如果页面文案、按钮、状态让你犹豫，请把这种犹豫本身记为证据。
- 如果你遇到阻断，先记录阻断，再尝试一条替代路径，而不是立刻结束。
- 你的目标是像认真又挑剔的真人用户一样，把真实使用中的阻力找出来。
